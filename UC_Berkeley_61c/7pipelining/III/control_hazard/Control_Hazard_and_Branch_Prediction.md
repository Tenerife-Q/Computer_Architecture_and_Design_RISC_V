# 掌握控制冒险：Control Hazard 与 分支预测 (Branch Prediction)

在流水线处理器中，除了数据因为前后依赖产生冲突外，流水线的“取指(IF)”本身也会面临迷茫。本文件夹重点剖析由 `Branch`（条件分支）和 `Jump`（无条件跳转）引发的 **Control Hazard (控制冒险)**。

## 1. 控制冒险的发生机制

在 5 级流水线中，由于 PC（程序计数器）在每个时钟周期都要 $+4$ 抓取下一条指令，所以 CPU 必须一直“闭着眼睛往下抓”。
但当我们遇到条件分支指令时：
```assembly
beq  x1, x2, Label   # 如果 x1 == x2，就跳到 Label 处执行
add  x3, x4, x5      # 分支不满足时执行的指令 1
sub  x6, x7, x8      # 分支不满足时执行的指令 2
...
Label:
or   x9, x10, x11    # 分支满足时执行的指令
```

**盲目的代价**：
如果是最原始的设计，`beq` 指令要在进入 **ALU (EX阶段)** 甚至是 **MEM阶段** 结束时，才知道到底条不跳转，以及跳转的目标地址是多少。
可是，在此等待的 2 到 3 个时钟周期内，流水线根本不能闲着，它已经默认 `PC = PC + 4`，强行把 `add` 甚至 `sub` 都抓进流水线并且解码了！

如果 `beq` 最终决定**“跳转到 Label”**，这时候流水线里已经充满了错误抓取的指令（`add` 和 `sub`）。

## 2. 硬件解法：冲刷清空 (Pipeline Flush / Zap)

如果抓错了怎么办？硬件的最后底线机制就是 **Flush (冲刷/清空)**。

当 `beq` 在 EX 阶段确认了分支条件满足，并且计算出了新地址：
1.  强势更新 `PC = 分支目标地址 (Label)`。
2.  迅速把目前在这条 `beq` 后面的、还在 IF 和 ID 阶段的指令全部**变成气泡 (NOP / Bubble)**——也就是把写状态位强行置 0，作废它们。

**代价**：这种设计的代价被称为 **Branch Penalty (分支开销)**。冲刷会白白浪费时钟周期，降低 CPU 性能。

## 3. 优化策略 1：提前计算 (Early Branch Resolution)

为了将 Flush 带来的损失降到最低，RISC-V 采取的架构优化是：**把分支决策前移**。
我们不要等到 EX 阶段用大 ALU 去算！我们可以在 **ID (译码阶段)** 增加一个专门的简单的比较硬件，同时在 ID 阶段提前计算目标地址。
这样，如果发现需要跳转，我们只需要 Flushing 收回一条刚好在 IF 阶段的指令，分支开销直接从 2 周期降低到 **1 周期**！

## 4. 优化策略 2：分支预测 (Branch Prediction)

即使只是 1 周期的 Flush 惩罚，在到处都是 `if-else` 和 `for` 循环的程序中也是巨大的性能灾难。
所以现代处理器都拥有算命一般的技能：**Branch Predictor (分支预测器)**。

1.  **静态预测 (Static Prediction)**：
    *   **Always Not Taken (总是预测不跳转)**：我就赌你顺序执行。猜对了全速前进，猜错了就 Flush（罚1拍）。
    *   **BTFNT (Backward Taken, Forward Not Taken)**：遇到往回跳的（比如 `for` 循环），预测它跳；往下面跳的（比如 `if` 避开一段代码），预测它不跳。
2.  **动态预测 (Dynamic Prediction)**：
    *   在 CPU 里内置一块高速缓存（BHT - Branch History Table，分支历史表）。
    *   它会悄悄记录某条分支指令上一次、上上一次是跳还是没跳（例如使用2位饱和计数器状态机）。
    *   取指时一旦遇到这条指令，直接查表看“天气预报”，然后按预测提前去拉取代码！现代 CPU 的分支预测成功率惊人地高达 **95% 以上**！

**总结**：控制冒险由跳转和分支引起。最笨的方法是等，保守的方法是提前在 ID 算完再 Flush 错的，最聪明的现代方法是依靠超高准确率的“分支预测”。
