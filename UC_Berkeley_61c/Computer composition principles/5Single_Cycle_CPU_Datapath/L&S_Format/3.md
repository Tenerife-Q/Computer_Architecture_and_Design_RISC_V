继续接着是L&S 这部分要求同上面

在理解了 R 和 I 类型的基础上，引入 **Load (加载)** 和 **Store (存储)** 指令是数据通路设计的又一重大飞跃。这标志着处理器正式具备了与**数据存储器 (Data Memory, D-Mem)** 交互的能力。

根据 CS61C Lecture 19.1 和 19.2 的内容，以下是对 L&S 部分的详细分析：

---

### 一、 Load 指令 (如 `lw`)：打通 WB 通路的最后一步
Load 指令（加载字）虽然在格式上属于 **I-Type**，但它在硬件执行逻辑上比 `addi` 复杂得多，因为它必须走完完整的五个阶段。

#### 1. 核心逻辑
Load 的任务是：计算地址 $\rightarrow$ 从内存读数据 $\rightarrow$ 写回寄存器。
$$Address = [rs1] + \text{sign\_ext}(\text{immediate})$$

#### 2. 硬件关键改动：Write-Back Mux
在之前的 R/I 类型中，写回寄存器堆（RegFile）的数据总是来自 ALU。为了支持 Load，我们在 RegFile 的输入端增加了一个 **WBSel (Write-Back Select) 多路选择器**：
* **选择 1 (ALU 结果)：** 用于普通的算术运算（如 `add`, `addi`）。
* **选择 0 (Data Mem 输出)：** 用于 Load 指令，将内存读取的结果送回寄存器。



---

### 二、 Store 指令 (如 `sw`)：S-Type 的独特性
Store 指令（存储字）将寄存器中的数据写入内存。它引入了两个全新的设计挑战：**不连续的立即数格式**和**内存写入路径**。

#### 1. S-Type 的格式挑战
Store 指令不需要 $rd$（目标寄存器），因为它写的是内存。为了保持 $rs1$ 和 $rs2$ 在指令中的位置固定（方便译码硬件），立即数被“拆散”到了指令的高 7 位和低 5 位中。
* **Imm Gen (立即数生成器)**：必须升级以处理这种分散的比特位。它会将指令中的 `inst[31:25]` 和 `inst[11:7]` 拼接并进行符号位扩展。

#### 2. 数据通路的改动
* **第二操作数路径：** ALU 仍然负责计算地址（$rs1 + offset$），因此 ALU 的 Input B 依然选择立即数。
* **数据输入端：** 寄存器 $rs2$ 的原始值（即我们要存入内存的数据）必须被直接引线到 **D-Mem 的 "Data In" 端口**。
* **控制信号：**
    * **MemRW：** 设置为 **Write (1)**，允许修改内存。
    * **RegWrite：** 必须设置为 **0**。这是关键！Store 指令不应该修改寄存器堆，虽然它占用了 $rd$ 的比特位，但我们通过关闭写使能来屏蔽它。



---

### 三、 L&S 阶段性对比总结

| 关键差异点 | Load (`lw`) | Store (`sw`) |
| :--- | :--- | :--- |
| **指令格式** | I-Type | **S-Type** (立即数拆分) |
| **ALU 角色** | 计算内存地址 | 计算内存地址 |
| **D-Mem 角色** | **Read** (输出到 RegFile) | **Write** (接收来自 $rs2$ 的数据) |
| **RegWrite** | 1 (需要更新 $rd$) | **0** (不更新寄存器) |
| **WBSel** | 选择来自 Memory 的值 | 无关 (Don't care) |

---

### 💡 深度洞察：单周期的瓶颈所在
在这一节的结尾，笔记暗示了一个重要的事实：**Load 指令是单周期 CPU 的“速度限制者”**。
由于单周期要求在一个时钟周期内完成所有操作，时钟频率必须设得足够慢，以容纳最长的执行路径：
> **PC $\rightarrow$ IMEM $\rightarrow$ RegFile $\rightarrow$ ALU $\rightarrow$ DMEM $\rightarrow$ WBMux $\rightarrow$ RegFile (Setup)**

这就是为什么 Load 指令需要完成所有 5 个阶段，它决定了整个处理器的性能上限。

接下来，当你开始研究 **B-Type (Branches)** 时，你会发现 PC 不再只是简单地 $+4$，而是会受到 ALU 比较结果的“左右”，那又是硬件逻辑的另一个有趣变数了。

### 一、 Load 指令 (以 `lw` 为例)：打通五级关卡

虽然截图主要是讲 Store，但我们先顺着你的逻辑补全 Load。在**图1 (R+I Arithmetic/Logic Datapath)** 中，你会发现最右侧的 DMEM (数据内存) 已经准备就绪了。

1. **计算地址 (ALU 的新兼职)：**
   对于 `lw x14, 8(x2)`，ALU 不再算单纯的数学题，而是做指针运算：`ALU = Reg[x2] + 8`。
2. **访存 (MEM 阶段登场)：**
   ALU 算出的地址，直接连到了 DMEM 的 `addr` 端口。此时控制逻辑给出 `MemRW = Read`，DMEM 根据地址吐出里面的数据 (`DataR`)。
3. **写回 (最右侧的终极 Mux)：**
   此时，最右侧的 **写回多路选择器 (WB Mux)** 终于派上用场了！控制信号 `WBSel = 1`（在图1中 0 代表 ALU，1 代表 Mem），让内存读出来的数据流向最上方的黄线，最终写回目标寄存器。
4. *(补充)* **关于图4 (All RV32 Load Instructions)：** 教授顺带提了一嘴，除了 `lw` (读 32 位)，还有 `lb` (读 8 位 byte) 和 `lh` (读 16 位 halfword)。它们是通过 `funct3` 区分的。在硬件底层，只需要在 DMEM 后面再加一点点掩码电路（切掉多余的位）和符号扩展电路即可。

### 二、 Store 指令 (`sw`)：RISC-V 架构设计的巅峰妥协

Store 指令是这节课最精彩的部分！请高度聚焦 **图5、图6 和 图8**。

#### 1. 为什么 S-Type 的格式这么“反人类”？(图5 & 图6)
对于 `sw x14, 8(x2)`，我们需要 3 个东西：基地址寄存器 `rs1` (x2)、源数据寄存器 `rs2` (x14)、和偏移量立即数 (8)。
你会发现 S-Type 没有 `rd` (目标寄存器)，这很合理，因为它不写寄存器。**但为什么要把 12 位的立即数硬生生劈成两半（高 7 位和低 5 位），分别放在指令的两端？为什么不连在一起？**

* **看图6的字幕：“在开始访问寄存器文件之前... 寄存器文件通常位于关键路径上。”**
* **架构师的权衡 (Trade-off)：** 如果为了立即数连续，把 `rs2` 的位置挪到原先 `rd` 的位置，那么 CPU 每次拿到指令，必须先花时间弄清楚：“这是 R 还是 S 指令？我是该读 20-24 位作为 rs2，还是读 7-11 位作为 rs2？”这会引入一个额外的 Mux 延迟。
* **RISC-V 的暴力美学：** 架构师规定，**无论什么指令，只要用到 rs1 和 rs2，它们在 32 位机器码中的位置绝对不准变！** (永远在 15-19 位和 20-24 位)。这样一来，取指完成后，硬件可以**闭着眼睛直接去读寄存器**，同时别的电路再去慢慢破译指令到底是什么。这极大地缩短了最慢的路径（Critical Path），提升了时钟频率。牺牲一点立即数拼接的复杂逻辑，换取全局读取速度，血赚！

#### 2. Imm Gen 的巧妙应对 (图8)
既然指令被拆了，硬件怎么拼接？看**图8 (I+S Immediate Generation)**。
其实没那么复杂。I 型和 S 型的立即数，高 20 位都是最高位的复制（符号扩展），而且第 11-5 位它们俩的位置也是完全重叠的！
**唯一的区别在于最低的 5 位 (位 4-0)：**
* I 型的低 5 位在 `inst[24:20]`。
* S 型的低 5 位在 `inst[11:7]`。
所以，硬件设计师只需要加一个**小小的 5 bit 多路复用器 (Mux)**（图8中间偏右那个梯形），就能完美兼容 I-Type 和 S-Type 的立即数生成！

### 三、 Store 的数据通路推演 (图2, 图3, 图7)

解决了指令格式，我们来看看电路上是怎么走的。对于 `sw`，请顺着高亮的黄线看：

1. **计算内存地址 (图3)：**
   `DataA` ($rs1$) 和扩展好的立即数 ($imm$) 进入 ALU 相加。**注意图3的字幕：“ALU的输出指向我们要写入这个数据的地址”**。这根黄线笔直地插进了 DMEM 的 `addr` 端口。
2. **数据的“暗渡陈仓” (图7)：**
   我们要存的数据在 $rs2$ 里。看图7中从 `Reg[rs2]` 出来的黄线。它虽然走到了 ALU 前面的 Mux，但 Mux 没有选它 (`BSel=1` 选了立即数)。
   **注意图7的字幕说明：** $rs2$ 的数据线在此处分叉了！一根绕过了 Mux，**直接连到了 DMEM 的 `DataW` (Data Write) 端口**（图7中最右侧亮起的黄线终点）。
3. **控制信号的最高指示 (图2)：**
   一切就绪，等待发令枪：
   * `MemRW = Write`：告诉内存，“敞开大门，把 DataW 端口的数据写到 addr 指向的格子里”。
   * `RegWEn = 0`：**至关重要！** 锁死寄存器的大门。因为 S 格式中 `inst[11:7]` 原本是 `rd` 的位置，现在被立即数占用了。如果不设为 0，CPU 就会把垃圾数据写进莫名其妙的寄存器。
   * `WBSel = *` (Don't Care，图2字幕解释了这点)：既然 `RegWEn` 都为 0 不写入了，最后那个 Mux 选 0 还是选 1 已经无所谓了。
