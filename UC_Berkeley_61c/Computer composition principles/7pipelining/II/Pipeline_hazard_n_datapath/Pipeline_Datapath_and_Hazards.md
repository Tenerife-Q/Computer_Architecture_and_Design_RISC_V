# 深入理解流水线设计 (二)：流水线数据通路与冒险概览

本文件分析如何从物理硬件层面将“单周期数据通路”改造成“流水线数据通路”，并引入阻碍流水线全速运行的核心问题——冒险 (Hazards)。

## 1. 硬件改造：流水线寄存器 (Pipeline Registers)

要让 5 个阶段能独立、并行地工作，互不干扰，必须在阶段与阶段之间加入“隔离墙”，这就是**流水线寄存器**。它们的作用是保存当前阶段运行得到的所有数据与控制信号，在时钟上升沿统一打拍子传递给下一阶段。

一共有 4 大“隔断”：
1.  **IF/ID Register**：保存取出的 32 位指令和当前 $PC$ 值。供 ID 阶段译码使用。
2.  **ID/EX Register**：保存从寄存器堆读出的两个数据 ($valA$, $valB$)，立即数 (Immediate)，目标寄存器编号 ($rd$)，以及**当前指令后续所有阶段需要的控制信号**（如 `ALUSel`, `MemRW`, `RegWEn` 等）。
3.  **EX/MEM Register**：保存 ALU 的计算结果，要存入内存的数据（对于 `sw`），目标寄存器编号 $rd$，以及后续 `MEM` 和 `WB` 阶段需要的控制信号。
4.  **MEM/WB Register**：保存从内存中读出的数据，ALU 的计算结果，目标寄存器编号 $rd$，以及写回信号 `RegWEn`。

### 1.1 控制信号的“击鼓传花”
在单周期中，控制信号瞬间传递给所有硬件。但在流水线中，如果 ID 阶段解码出现在是一条 `lw` 指令，它的 `MemRead` 信号绝对不能立刻送到 Memory！因为此时 Memory 正在为前几条指令服务。
因此，`lw` 的 `MemRead` 信号必须被存入 ID/EX，下个周期传到 EX/MEM，直到到达 MEM 阶段时才被激活生效。

## 2. 流水线的最大挑战：冒险 (Hazards)

当多条指令同时在一个处理器中奔跑时，它们极易产生冲突。这种导致下一条指令无法在预定周期内正确执行的现象，称为 **Hazard (冒险)**。

主要分为三大类：

### 2.1 结构冒险 (Structural Hazard)
**定义**：硬件资源不够，两条指令在同一个周期争抢同一个组件。
**经典案例**：
*   **内存冲突**：如果 CPU 只有一个内存模块，IF 阶段要读指令，MEM 阶段要读写数据，两者都在抢内存。
    *   *解法*：哈佛架构 (Harvard Architecture)——将内存从物理上分为**指令缓存 (Instruction Cache)** 和**数据缓存 (Data Cache)**。
*   **寄存器堆冲突**：WB 阶段要往寄存器写数据，ID 阶段要从寄存器读数据。
    *   *解法*：改变寄存器堆的设计，让它在一个时钟周期的**前一半写入 (Write on falling edge) ，后一半读取 (Read on rising edge)**。这样同一周期对同一寄存器的读写也能拿到最新值。

### 2.2 控制冒险 (Control Hazard)
**定义**：在遇到分支 (Branch) 或跳转 (Jump) 指令时，CPU 必须等算清楚了“跳转条件”和“目标地址”才能决定下一条指令去哪。但在判断清楚之前，流水线已经瞎抓了好几条指令进来了。
*   *解法简介*：插入停顿 (Stall) 直到条件明确，或者使用高级的**分支预测 (Branch Prediction)**。猜对了继续跑，猜错了就清空流水线 (Flush) 重新来。

### 2.3 数据冒险 (Data Hazard)
**定义**：后一条指令需要用到前一条指令即将算出、但还没写回的数据。这是下一节文档的重点！

## 3. 隐形的核心点位：状态同步与控制流

### 3.1 控制信号的“流水化” (Control Signal Pipelining)
在单周期 CPU 中，控制器根据指令直接生成所有控制信号。但在流水线中，控制信号是在 ID（译码）阶段一次性生成的，但它们并不会立即全部生效。
*   **实现机制**：控制信号就像随身行李一样，存放在流水线寄存器中，随着指令一起在各阶段之间“接力”。例如，`MemWrite` 信号会保存在 ID/EX、EX/MEM 寄存器中，直到指令到达 MEM 阶段才被真正使用。

### 3.2 PC 值的“接力赛”
PC 值像指令本身一样，存储在流水线寄存器中，从 IF/ID 开始一路向右传递。
**为什么 WB 阶段还需要知道 PC？**
1.  **异常与中断处理 (Exceptions)**：如果指令在 MEM 或 EX 阶段发生了错误（如内存越界），处理器需要将该指令的 PC 存入 EPC (Exception Program Counter)。如果没有随行传递，就不知道是哪条指令出了错。
2.  **JAL/JALR 指令**：跳转链接指令需要将 `PC + 4` 写回到寄存器堆，写回发生在 WB 阶段。
3.  **PC 相对寻址指令 (AUIPC)**：需要将 PC 与立即数相加并在 WB 阶段写回。

### 3.3 寄存器堆的内部“时序转换” (Double Pumping)
为了解决 WB 阶段写寄存器和 ID 阶段读同一个寄存器的**结构冒险**，硬件做了一个巧妙的平衡：
*   **前半个时钟周期写 (Write on falling edge/first half)**。
*   **后半个时钟周期读 (Read on rising edge/second half)**。
这样在同一个时钟周期内，WB 写回的数据在后半周期就能被 ID 阶段读到，在物理底层消除了这种特定情况的数据冲突。

## 4. 巩固练习：结构冒险与数据通路理解

**题目 1：流水线寄存器宽度（原理题）**
*   **问题**：在五级流水线中，ID/EX 流水线寄存器至少需要保存哪些信息，才能保证后续阶段正常工作？
*   **解答**：这些寄存器是递给下一阶段的“快递盒”：
    1.  **数据类**：`RS1_Data`, `RS2_Data`（从 RegFile 读出的值）、`Imm`（符号扩展后的立即数）。
    2.  **控制类**：`ALUSel`（ALU 做什么运算）、`ALUSrc`（选择立即数还是寄存器值）等。
    3.  **元数据类**：`PC` 值、`RD_Index`（目标寄存器编号，留到 WB 阶段写回用）。

**题目 2：结构冒险（概念判断）**
*   **问题**：为了省钱，我们将指令内存 (IMEM) 和数据内存 (DMEM) 合并为一个单一接口的内存（同一时间只能读或写一次）。哪两个流水线阶段会发生结构冒险？
*   **解答**：**IF (取指令) 和 MEM (访存)**。当流水线中有一条 `lw` 指令处于 MEM 阶段读数据时，同时必须有一条新指令在 IF 阶段尝试读取指令码。接口唯一导致硬件冲突。这就是为什么现代 CPU 必须拥有独立的 L1 I-Cache 和 L1 D-Cache。
