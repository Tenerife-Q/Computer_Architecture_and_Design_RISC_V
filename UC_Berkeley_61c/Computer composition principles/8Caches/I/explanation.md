### 突破口：Dan 教授速算法的“底层数学原理”

在计算机世界里，所有的地址线、内存容量都是以 **2 的幂次方** ($2^n$) 来计算的。但是，人类的大脑习惯于**10 的倍数**（千、百万、十亿）。

这套速算法的本质，就是在**二进制（计算机）和十进制（人类认知）之间搭建一座快速翻译的桥梁**。

#### 1. 唯一的基石定理：$2^{10} \approx 10^3$
一切魔法都来源于这个巧合：
* $2^{10} = 1024$
* $10^3 = 1000$
它们非常接近！为了严谨，计算机科学家发明了专门的词汇来指代 $2^{10}$ 的倍数，也就是 **Kibi, Mebi, Gibi, Tebi** (后缀加了个 bi，代表 binary)。

#### 2. “拆分指数”的数学推导
假设我们要计算 $2^E$（$E$ 是一个两位数）。
在数学上，任何一个两位数 $E$ 都可以拆成：**$10 \times x \ + \ y$**
> （其中 $x$ 是十位数字，$y$ 是个位数字。例如 34，十位 $x=3$，个位 $y=4$，即 $10 \times 3 + 4$）。

所以，$2^E$ 就可以写成：
$2^{10x + y}$

根据高中的指数运算法则（$a^{b+c} = a^b \times a^c$），我们可以把它拆开：
$2^{10x + y} = \mathbf{(2^{10})^x \times 2^y}$

这就是 Dan 教授那个“速算法”的**绝对核心公式**！我们来看看它是怎么和那几条规则对应上的：

* **看公式右半边 $\mathbf{2^y}$ （个位决定基础数值）：**
  $y$ 是指数的个位，由于个位只能是 0~9，所以你只需要熟记 $2^0$ 到 $2^9$ 的值（1, 2, 4, 8, 16, 32, 64, 128, 256, 512）。这就计算出了前面的**数字大小**。

* **看公式左半边 $\mathbf{(2^{10})^x}$ （十位决定单位前缀）：**
  刚才我们说了 $2^{10}$ 就是 Kibi。
  * 当十位 $x=1$ 时：$(2^{10})^1 = 2^{10}$ $\rightarrow$ **Kibi** (千)
  * 当十位 $x=2$ 时：$(2^{10})^2 = 2^{20}$ $\rightarrow$ **Mebi** (兆/百万)
  * 当十位 $x=3$ 时：$(2^{10})^3 = 2^{30}$ $\rightarrow$ **Gibi** (吉/十亿)
  * 当十位 $x=4$ 时：$(2^{10})^4 = 2^{40}$ $\rightarrow$ **Tebi** (太/万亿)

#### 3. 重新实战演练（带着数学原理）
现在我们再来看 $2^{34}$，你的大脑里应该自动弹出这个推导过程：
1. **数学拆解**：$2^{34} = 2^{30 + 4} = (2^{10})^3 \times 2^4$
2. **算个位（数字）**：$2^4 = 16$
3. **算十位（单位）**：$(2^{10})^3 \rightarrow$ 3 次方对应 Gibi
4. **拼接**：16 Gibi。

**反向计算（从容量推导位数）也一样简单**：
如果题目说：“我有一个 4 Gibi 的内存，需要多少位地址线？”
1. **翻译数字**：4 是 $2^2$（个位 $y=2$）。
2. **翻译单位**：Gibi 是 $(2^{10})^3$ 即 $2^{30}$（十位 $x=3$）。
3. **拼接**：总指数就是 $30 + 2 = 32$。所以需要 **32 bits** 的地址线。

*（懂了这个数学推导，你就再也不用死记硬背了，因为你掌握了它为什么是这样的规律。）*

### Caches I 的核心概念：我们为什么需要缓存？

搞定了算数，我们再来看概念。Caches I 只需要你建立一个核心认知：**造物主的妥协与人类的欺骗术**。

#### 1. 内存墙 (The Memory Wall)
* **SRAM（做寄存器、Cache 的材料）**：速度极快（能跟上 CPU），但贼贵，且占地极大。所以 CPU 里面只能塞下一点点。
* **DRAM（做内存条的材料）**：便宜，容量极大，但速度慢得令人发指（对 CPU 来说）。
**矛盾**：CPU 运算只要 1 纳秒，但去主存拿数据要 100 纳秒。CPU 有 99% 的时间都在喝茶等数据。
**目标**：我们想要一个**“既像 SRAM 一样快，又像 DRAM 一样大”**的存储器。物理上不可能实现，所以计算机科学家用 Cache 创造了一个**“幻觉 (Illusion)”**。

#### 2. 幻觉的支柱：局部性原理 (Principle of Locality)
为什么只要把最常用的那一小撮数据放在 Cache 里，就能骗过 CPU，让它觉得整个巨大的内存都很快？因为人类写的程序极其缺乏想象力，高度符合两个规律：

* **时间局部性 (Temporal Locality)**：
  * **定义**：如果一个数据刚才被访问了，它**在不久的将来极有可能再次被访问**。
  * **代码案例**：`for (int i = 0; i < 1000; i++) { sum += i; }` 中的 `sum` 和 `i`。在极短时间内被反复读取和修改。
  * **Cache 的应对**：CPU 用过的数据，别急着扔掉，放进 Cache 里留一会儿。

* **空间局部性 (Spatial Locality)**：
  * **定义**：如果一个数据被访问了，它**物理地址相邻的其他数据也极有可能很快被访问**。
  * **代码案例**：遍历一个巨大的数组 `arr[0], arr[1], arr[2]...`。数组在内存中是连续存放的。
  * **Cache 的应对**：这就是为什么 Cache 有 **Block (块)** 的概念。CPU 哪怕只要 `arr[0]` 的 1 个字节，Cache 也会顺手把紧挨着的 `arr[1]` 到 `arr[15]` 打包成一个 Block 全部搬进 Cache。当 CPU 接着要 `arr[1]` 时，直接在 Cache 中**命中 (Hit)**，省去了去主存搬运的时间。

#### 3. 图书馆比喻（最经典的理解方式）
* **主存 (DRAM)**：学校的巨大地下图书馆。书多，但下去找一本书要花 1 个小时。
* **Cache**：你书房的书桌。很小，只能放几本书，但一伸手就能拿到。
* **时间局部性**：你正在写关于 CS61C 的论文，那本《计算机组成与设计》你今天翻了，明天大概率还要翻。所以别还不回地下室，留在书桌上。
* **空间局部性**：你需要查阅某一页的内容。图书管理员（硬件）不会把那一页撕下来给你，而是会把**整本厚厚的书（Block）**搬到你的书桌上。因为你看了第 100 页，大概率马上就会看第 101 页。

**你的大脑现在已经准备好将这两个知识点结合了：**

因为内存实在太大了（比如几 Gibi），我们需要用**二进制切分地址（T.I.O）**来追踪这些 Block。
