### 第一步：拆解缓存架构与地址 (The Setup)
*(对应图中 `Accessing data in a direct mapped cache` 讲解 16KB、4 word blocks 的那两张图)*

在玩转 Cache 之前，我们必须先知道这块 Cache 的“三围”尺寸，并据此将 32 位内存地址切分成著名的 **T:I:O (Tag : Index : Offset)**。

根据 PPT 的设定：
1.  **Block Size (块大小):** 4 words = 16 Bytes。所以我们需要 **4 bits** 来作为 **Offset（偏移量）**，因为 $2^4 = 16$，这样才能精确指到这 16 个字节中的某一个。
2.  **Cache Size (总容量):** 16 KB。
3.  **Index (索引位):** 既然总容量是 16KB，每个块是 16B，那么一共有 $16KB / 16B = 1024$ 行（Blocks）。为了能索引 1024 个行，我们需要 **10 bits** 来作为 **Index（门牌号）** ($2^{10} = 1024$)。可以看到 PPT 左侧的 Index 正是从 `0` 一直标到了 `1023`。
4.  **Tag (标签位):** 剩下的位数 $32 - 10 - 4 =$ **18 bits**。

**关键动作：**
当 CPU 想要访问四个地址 (`0x00000014`, `0x0000001C`, `0x00000034`, `0x00008014`) 时，硬件的第一反应就是把这些 16 进制数字转成二进制，然后像切蛋糕一样切成 `Tag | Index | Offset`（如你图中黑底蓝绿红字的 PPT 所示）。

### 第二步：发生未命中与替换 (Miss & Replace)
*(对应标题为 `Cache Block 1 Tag does not match` 和 `Miss, so replace block 1...` 的两张图)*

这就是前面提到的“看门牌号入座”。我们来看看当 CPU 请求读取 `0x00008014` 时发生了什么：

1.  **拆分地址：** CPU 将 `0x00008014` 拆分后发现，**Index 是 `1`，Tag 是 `2`**。
2.  **对号入座：** 硬件立刻跑去看 Cache 的第 `1` 行（Index 1）。
3.  **检查匹配：** Cache 的第 `1` 行此时是有数据的（Valid = 1），里面装的是数据 `a, b, c, d`。但是！硬件一看这行的 **Tag 是 `0`**。
4.  **判定结果：** **Tag 2 $\neq$ Tag 0**。你要找的人不在，这是一个标准的 **Cache Miss**！
5.  **残酷的替换 (Replace)：** 因为是直接映射（Direct Mapped），`0x00008014` 这个内存块**只能**放在 Index 1 这个位置。所以它别无选择，必须把原本占据这里的 `a, b, c, d` 无情地踢走，然后去主存（Memory）把自己的数据 `i, j, k, l` 搬进来，并将这一行的 **Tag 更新为 `2`**。

> **💡 知识点链接：** 注意看，Cache 中 `0, 2, 4, 5...` 那么多行全是空的！但就因为它们映射到了同一个 Index 1，导致了互相驱逐。这就是最典型的 **Conflict Miss（冲突未命中）**！

### 第三步：教授的随堂测验 (Do an example yourself & Answers)
*(对应 `Do an example yourself...` 和 `Answers` 这两张图)*

现在 Cache 里的状态更新了（Index 1 变成了 Tag 2，数据是 `i,j,k,l`；Index 3 是 Tag 0，数据是 `e,f,g,h`）。教授抛出了两个地址让你判断：

**测试 1：读取地址 `0x00000030`**
*   **拆解：** 二进制拆分后，**Index = 3, Tag = 0**, Offset = `0000` (指向第0个字节，即该块最右边的字)。
*   **找位置：** 去 Cache 的 Index 3 看一眼。
*   **比对：** Valid 是 1，Tag 是 0。**0 == 0，匹配成功！** 这是一次 **Cache Hit (命中)**。
*   **取数据：** 根据 Offset 取出对应的数据，返回了 `e`。

**测试 2：读取地址 `0x0000001C` （高能预警！）**
*   **拆解：** 二进制拆分后，**Index = 1, Tag = 0**, Offset = `1100` (十六进制的 0xC，指向该块最左边的字)。
*   **找位置：** 去 Cache 的 Index 1 看一眼。
*   **比对：** Index 1 现在的 Tag 是多少？是刚才替换进来的 **`2`**！但我们现在需要的是 Tag **`0`**。
*   **结果：** **Tag 0 $\neq$ Tag 2**。再次发生 **Cache Miss**！
*   **后果：** 没办法，只能再次去主存把 `a, b, c, d` 请回来，把刚坐热乎的 `i, j, k, l` 踢出去。返回数据 `d`。

### 总结：这几张 PPT 告诉了我们什么？

Dan Garcia 教授通过这个精妙的例子，不仅让你亲手推演了 T:I:O 的计算和寻址过程，更揭示了 Directly Mapped 架构最致命的弱点：**乒乓效应 (Ping-Pong Effect)**。

如果你在代码里交替读取 `0x00008014` 和 `0x0000001C`，你的 Cache 命中率将会是凄惨的 **0%**。虽然 Cache 容量有 16KB，但因为这两个地址的 **Index 相同**，它们会疯狂地把对方踢出去，导致 Cache 彻底失去意义。

这也是为什么在下一节课中，CS61C 会引入 **Set-Associative（组相联）**——给每个 Index 分配两个或更多的“座位”，来缓解这种撞车惨剧。
