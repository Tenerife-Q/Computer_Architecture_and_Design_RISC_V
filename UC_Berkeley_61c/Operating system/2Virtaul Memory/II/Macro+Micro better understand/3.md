### 模块十一：流水线里的“左右护法”（为什么有两个 TLB？）

在经典的五级流水线（取指 -> 译码 -> 执行 -> 访存 -> 写回）中，TLB 到底站在哪里？

**1. 现象：CPU 里其实有两个 TLB！**
*   **左护法（指令 TLB，Instruction TLB）：** 站在流水线的最前端（第 1 级）。
*   **右护法（数据 TLB，Data TLB）：** 站在流水线的访存端（第 4 级）。

**2. 为什么要搞两个？（防止“抢车道”）**
*   CPU 每执行一条指令，首先要**去内存把指令代码取出来**，这需要翻译地址（交给左护法）。
*   如果这条指令恰好是 `lw`（读取数据），CPU 还需要**去内存把数据拿出来**，这也需要翻译地址（交给右护法）。
*   如果整个 CPU 只有一个 TLB，取指令和拿数据就会在同一个时钟周期内“打架”（这在体系结构里叫**结构冲突 Structural Hazard**）。所以，必须把指令 TLB 和数据 TLB 物理分离开！

---

### 模块十二：查表时的“三颗定时炸弹”（硬件 vs 操作系统的分工）

当虚拟地址被送进 TLB 时，可能会随时引爆下面这三颗炸弹。**注意看是谁来拆弹的！**

**💣 第一颗炸弹：TLB Miss（小抄上没找到）**
*   **触发原因：** TLB 里没有这条地址的翻译记录。
*   **拆弹专家：硬件查表员（Hardware Page Table Walker）。**
    *   在现代 CPU 里，这颗炸弹**不需要惊动操作系统**。
    *   CPU 内部有一个纯硬件的状态机，它会自动顺着 `SPTBR` 寄存器，跑到主存里去翻 L1、L2 页表。找到后，自动把记录写进 TLB，然后让 CPU 重新执行刚才的动作。
    *   **代价：** 几十到上百个时钟周期（慢，但能接受）。

**💣 第二颗炸弹：Page Fault（缺页异常 - 字典里写着“在硬盘上”）**
*   **触发原因：** 硬件查表员在主存里找到了 PTE，但发现 `Valid = 0`（且不是越界），说明这页数据当初因为主存满了，被踢到慢吞吞的**磁盘（Disk）**上去了！
*   **拆弹专家：操作系统（OS）。**
    *   硬件彻底搞不定了，触发**精确异常 (Precise Trap)**，强行暂停当前程序，呼叫 OS 大哥接管。
    *   OS 发送指令，让磁盘把那页数据慢慢运回主存。
    *   **代价：** 两千万个时钟周期！（极其惨痛）。

**💣 第三颗炸弹：Protection Violation（保护违规 / 越权访问）**
*   **触发原因：** 找到了 PTE，但发现 `R/W/X` 权限不对！比如一个普通用户程序，非要去写操作系统的只读内核代码；或者数组越界，访问了不属于自己的内存。
*   **拆弹专家：操作系统（OS）的“死刑判决”。**
    *   硬件立刻拦截，报告 OS。
    *   OS 看到后，绝对不留情面，直接把这个违规的程序**强制杀掉**！
    *   **现象：** 这就是你在写 C 语言时，最臭名昭著的报错——**`Segmentation Fault (Segfault) 段错误`**！

### 模块十三：狸猫换太子——Context Switch（上下文切换）的代价

为了让你觉得微信和网易云在“同时”运行，操作系统每隔几十毫秒就会在它们之间切换，这就叫**上下文切换 (Context Switch)**。

**1. 切换时的致命漏洞：**
*   准备从“微信”切换到“网易云”。
*   OS 更改了 `SPTBR` 寄存器，让它指向网易云的 L1 页表。
*   **危险：** 此时 CPU 的 TLB 里，存的全是**微信的地址翻译记录**！如果不加处理，网易云一跑起来，查 TLB 命中的全是微信的物理内存。网易云就能直接偷看微信的聊天记录，系统大乱！

**2. 强硬的补救措施（TLB Flush）：**
*   在切换的瞬间，OS 必须执行一条极其暴力的指令：**把 TLB 里的所有记录全部清空（设为 Invalid）！**
*   这叫做 **TLB Flush（刷新快表）**。

**3. 切换后的“阵痛期”（冷启动）：**
*   网易云刚开始运行的前几百条指令，每一次去查 TLB，都会发现是空的（TLB Miss）。
*   此时系统会疯狂触发**第一颗炸弹**，硬件查表员疲于奔命地去主存查表。
*   所以，程序刚切换过来的那一瞬间，**运行速度是极慢的**。直到网易云把常用的地址重新填满了 TLB（这叫 TLB 预热），速度才会重新起飞。

### 模块十四：缺页异常时的“时间管理大师”（OS 的摸鱼哲学）

我们刚才说到，触发**第二颗炸弹（Page Fault）**时，OS 要去磁盘搬数据，需要整整**两千万个时钟周期**。

在这漫长的几毫秒里，CPU 难道就傻站着等吗？
**绝对不行！OS 是个极致的“时间管理大师”。**

*   **OS 的骚操作：**
    1. OS 吩咐硬盘控制器：“你去把微信需要的那一页数据给我搬到内存来，搬好了叫我。”
    2. OS 立刻对 CPU 说：“微信现在卡住了，别等它了。**立刻进行上下文切换 (Context Switch)**，把网易云调进来跑！”
    3. 于是，在硬盘搬运微信数据的这段时间里，CPU 一直在欢快地播放网易云音乐。
    4. 过了几毫秒，硬盘喊了一声：“数据搬完了！（触发中断）”
    5. OS 收到通知，更新微信的页表，然后再找个时机把 CPU 切回微信。微信的指令重新执行，这一次完美命中！
