### 模块一：为什么必须用“多级”页表？（解决空间危机）

**1. 灾难现场（单级页表）：**
*   **规则：** 单级页表就像一本字典，无论字存不存在，都必须把 104 万个词条（$2^{20}$）全印出来。
*   **代价：** 1 个进程占 **4MB** 内存；256 个进程占 **1GB** 内存。
*   **浪费：** 程序的内存中间有巨大的空洞（甜甜圈洞），这部分明明没用到，但字典里还是给它留了占地方的空白行。

**2. 解决方案（多级页表）：**
*   **策略：** 把 1 本大字典，拆成“**1 本总目录 (L1)** + **很多本分册 (L2)**”。
*   **神级操作（划重点）：** 如果总目录 (L1) 发现某一块地址属于“空洞”（没人用），它会直接标记为无效。**系统就不会去物理内存里印那本对应的分册 (L2)！**
*   **结果：** 完美避开了空洞，原本 4MB 的页表，现在可能只需要 **12KB**（1页L1 + 2页L2）。

### 模块二：硬核推导：32位虚拟地址怎么切？

在 RISC-V 32位系统中，虚拟地址被严格切分为：`10 位 + 10 位 + 12 位`。
这不是随便定的，下面是**极其严密的倒推计算过程**：

**▶ 第 1 步：确定“页内偏移量 (Offset)”**
*   **已知条件：** 操作系统规定 1 个页面大小为 **4KB**。
*   **单位换算：** $4\text{KB} = 4096 \text{ 字节}$。
*   **计算：** 要在一个 4096 字节的空间里寻址，需要多少位？因为 $2^{12} = 4096$，所以需要 **12 位**。
*   **结论：** 32位地址中，**最后 12 位**被固定为 Offset。剩下的 20 位用来做页号。

**▶ 第 2 步：确定“一本字典”能装多少条目？**
*   **已知条件：** 工程师希望每一级页表（小字典）的大小，**刚好占据 1 个物理页（4KB = 4096 字节）**，方便管理。
*   **已知条件：** 字典里的一条记录（叫做 PTE）占 **4 字节**。
*   **计算：** $4096 \text{ 字节 (一页)} \div 4 \text{ 字节 (一条)} = 1024 \text{ 条}$。
*   **结论：** 一本页表刚好能写下 **1024** 行记录。

**▶ 第 3 步：确定“页表索引 (Index)”的位数**
*   **计算：** 要在一个包含 1024 行记录的本子里查找某一行，需要几位二进制数？
*   **答案：** 因为 $2^{10} = 1024$，所以需要 **10 位**。
*   **结论：** 剩下的 20 位虚拟页号，被完美地一分为二，变成了 **10 位 (查 L1)** 和 **10 位 (查 L2)**。

> **📌 最终地址结构图：**
> `[ L1 索引 (10 bits) ]` | `[ L2 索引 (10 bits) ]` | `[ 偏移量 Offset (12 bits) ]`

### 模块三：字典里的内容：PTE 长什么样？

找到了那行记录（PTE，页表条目），这 4 个字节（32 bits）里到底存了什么？

| 字段名称 | 占用位数 | 功能说明 |
| :--- | :---: | :--- |
| **PPN** | 22 bits | **物理页号**。翻译结果，告诉你真实数据/下一级页表在哪里。 |
| **RSW** | 2 bits | 留给操作系统软件自己用的，硬件不管。 |
| **D, A, G, U**| 4 bits | 脏位、已访问位、全局位、用户态权限位。 |
| **X, W, R** | 3 bits | **执行(X)、写(W)、读(R)权限。** (判断是否到头的关键) |
| **V** | 1 bit | **有效位(Valid)**。1=有效，0=无效(缺页/越界)。 |

**⚠️ 硬件如何判断查完没有？（核心判断逻辑）：**
*   **如果 `R=0, W=0, X=0`：** 硬件一看，没有任何读写权限。它就知道：**这不是数据！这是一个指针，PPN 指向的是下一级 L2 页表！**
*   **如果 `R/W/X` 哪怕有一个是 1：** 硬件就知道：**翻译到底了！这是一个叶节点，PPN 指向的就是最终真实的物理数据页！**

### 模块四：查表流水线（一步步怎么找数据）

最后，我们把上面三个模块串起来，看看硬件是怎么跑完这趟流程的：

1.  **起始：** CPU 读到一个寄存器叫 `SPTBR`，获取 **L1 总目录的物理基地址**。
2.  **查 L1：** 截取虚拟地址的高 **10位 (`p1`)**，去 L1 总目录里找到第 `p1` 行 (PTE)。
    *   *假设 PTE 有效，且 RWX=0。*
3.  **查 L2：** 硬件提取该 PTE 里的物理页号，找到了 **L2 分册的位置**。
    *   截取虚拟地址的中间 **10位 (`p2`)**，去 L2 分册里找到第 `p2` 行 (PTE)。
    *   *假设 PTE 有效，且 RWX 不为 0。*
4.  **拿数据：** 硬件提取该 PTE 里的物理页号（最终数据的地址）。
    *   把它和虚拟地址最后的 **12位 (Offset)** 拼接起来。
    *   拿着这个最终地址，去 Cache / 主存拿真实包裹！

多级页表的核心目的不是因为“L1单页装不下”，而是为了**“允许大部分 L2 页表根本不在物理内存中被创建”**，从而实现空间压缩。

你的前置数学换算（4KB页 $\div$ 4字节PTE = 1024条记录 = 10 bits索引）已完全通过逻辑验证。直接进入物理映射的本质转换：

### 核心解答：L1 与 L2 字段的绝对物理指向
1. **L1 的 10 bit（寻找目录分册）：**
   在固定的 L1 页表（总目录）中索引 1024 个条目。**这里的每一个 PTE，指向的是某一张独立 L2 页表（分册）的物理基地址。**
2. **L2 的 10 bit（寻找数据页）：**
   当通过 L1 找到具体的一张 L2 页表后，这 10 bit 用于在该 L2 页表中索引 1024 个条目。**这里的 PTE，指向的是最终存放真实数据的物理页基地址。**

### 为什么必须分层？（利用稀疏性进行物理截断）
* **单层架构的致命缺陷：** 必须在主存中分配连续的 4MB 物理空间，一次性硬编码 $2^{20}$（104万）个 PTE。即使程序中间有 99% 的空间未被使用（甜甜圈空洞），这 4MB 的物理内存也被强行锁死。
* **双层架构的终极解法：** L1 页表中的一个条目，掌控着背后连续的 4MB 虚拟地址空间（1024 个 L2 条目 × 4KB 页）。如果程序根本没访问这 4MB 空间，**L1 中的该条目直接标记为无效（Valid=0）。此时，系统绝对不会在物理内存中为它分配那张占 4KB 空间的 L2 页表。**

最终结果：一个简单的程序，物理内存中只存在 1 张 L1 页表和 2 张 L2 页表（共占 12KB），完美去除了单级页表强行占据的 4MB 冗余。
