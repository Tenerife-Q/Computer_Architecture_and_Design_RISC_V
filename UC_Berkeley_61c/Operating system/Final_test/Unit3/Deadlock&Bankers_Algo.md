### 补充考点 1：时间片轮转调度 (Round Robin, 简称 RR)

这是期末考试的常客，它的核心逻辑就是一个词：**“排队吃果果，一人一口，吃不完重新排”**。

**📌 绝对规则：**
1. 系统会设定一个固定的时间段，叫做**时间片 (比如 q=2)**。
2. 进程只要拿到 CPU，**最多**只能运行 2 个时间单位。
3. 如果 2 个单位用完了，但进程还没执行完，系统会强行把它**踢到就绪队列的最尾部**重新排队。
4. 如果进程需要的剩余时间 < 时间片（比如只需要 1 秒），那它执行完这 1 秒后就直接结束，CPU 立刻分给下一个人，**不用干等**。

**📝 实战推导 (还是用我们熟悉的 A、B、C)：**
假设：时间片 **q = 2**。
*   A 到达:0，服务:5
*   B 到达:1，服务:2
*   C 到达:2，服务:1

**执行时间轴（注意看每次排队的情况）：**
*   **t=0 时刻**：A (到得最早) 占用 CPU。
*   **t=1 时刻**：B 到达。B 在旁边排队。
*   **t=2 时刻**：
    *   C 到达。C 在 B 后面排队。
    *   同时，**A 的时间片 (2) 用完了！** A 还剩 3 (5-2=3)。系统强行把 A 剥夺，放到队伍最后。
    *   *此时排队情况：队首 B -> C -> A 队尾*
    *   轮到队首的 **B 占用 CPU**。
*   **t=4 时刻**：
    *   B 运行了 2 个单位，**B 刚好运行完！** (因为 B 总共就需要 2)。
    *   *此时排队情况：队首 C -> A 队尾*
    *   轮到队首的 **C 占用 CPU**。
*   **t=5 时刻**：
    *   C 只需要 1 个单位时间。此时 C 运行完毕！**直接交出 CPU** (不用等满 2 秒)。
    *   *此时排队情况：队首 A 队尾*
    *   轮到 **A 占用 CPU**。
*   **t=7 时刻**：
    *   A 运行了 2 个单位，时间片又用完了！A 还剩 1 (3-2=1)。
    *   由于没有别人排队，A 自己排自己后面，**A 继续占用 CPU**。
*   **t=8 时刻**：
    *   A 运行完最后 1 个单位，**A 彻底结束**。

*(你自己在纸上画甘特图，只要标出这段时间是谁在运行，找出最后结束的时间，就能算出周转时间。这题非常考验细心)*

### 补充考点 2：优先级调度

这个考点往往不单独出大题，而是和上面的算法结合。你只要记住：**不要去管它们运行需要多长时间，只看优先级数字！**

**📌 考试避坑指南：**
考试卷子上一定会有一列写着“优先级”。
第一步：**先读大题题干！** 看清楚括号里写的是“数字越大优先级越高”还是“数字越小优先级越高”。(国内大部分教材默认是数字**越大**优先级**越高**，但 UNIX 系统里 0 是最高优先级，所以一定要看题)。
第二步：看题干写的是**抢占式**还是**非抢占式**。
*   **非抢占**：高优先级的来了，也得在旁边乖乖等当前的人运行完。
*   **抢占**：高优先级的来了，立刻一脚把当前运行的人踢走，自己上 CPU。

### 👑 第三阶段：死锁 与 银行家算法 (Banker's Algorithm)

这就进入到了你大纲里的压轴部分（★★级别）。银行家算法通常是卷子上最后一道、分值最大的计算题（10~15分）。

为了不让你觉得突兀，这里稍微提一句背景：为什么叫“银行家”算法？因为这个算法最早就是为了解决银行贷款问题的（操作系统里的“资源”就像银行里的“钱”，“进程”就像“借钱的客户”）。银行不能把钱一次性全借给那些还不上的人，否则银行就破产了（死锁）。

#### 📌 知识点一：必背的 4 个矩阵/表格名字

在做银行家算法的题时，题目一定会给你一张表。表里有四个核心的数据概念，你必须深刻理解它们的意思：

1.  **Max（最大需求矩阵）**：这个进程**总共**需要多少资源才能干完活。*(大白话：客户总共想借多少钱)*
2.  **Allocation（已分配矩阵）**：系统**已经**给了这个进程多少资源。*(大白话：银行已经借给客户多少钱)*
3.  **Need（需求矩阵）**：这个进程**还需要**多少资源。
    *   **⚡ 必记公式：`Need = Max - Allocation`** *(大白话：还需要借的钱 = 总共想借的钱 - 已经借到的钱)*
    *   *注：考试时通常不给你 Need 矩阵，你需要自己在草稿纸上用减法算出来！*
4.  **Available（可用资源向量）**：系统当前**手里还剩下**多少空闲资源。*(大白话：银行金库里现在还有多少钱)*

#### 📌 知识点二：安全性算法（找“安全序列”）

考试最常见的问法第一题：“**请问当前系统是否处于安全状态？如果安全，请给出一个安全序列。**”

**解题逻辑（也就是你在草稿纸上的操作步骤）：**
*   **目标**：你要在当前的系统里，找出一个进程，满足条件：**它需要的资源 (Need) $\le$ 系统手里剩下的资源 (Available)**。
*   **动作**：如果满足，说明系统剩下的资源足够让这个进程干完活。你就假设把资源借给它。
*   **结果**：它干完活之后，不仅会把刚才借的资源还回来，还会把它**以前占用的资源 (Allocation) 全部还给系统**！
*   **系统资源更新**：**新 Available = 原 Available + 该进程的 Allocation**。
*   **重复**：系统资源变多了，你再去寻找下一个能满足的进程，直到所有进程都顺利干完活。
*   **结论**：如果你能找到一个顺利干完活的顺序（比如：P1 -> P3 -> P0 -> P2），这就是**安全序列**。系统就是安全的。

#### 📝 实战带练：寻找安全序列

我们来看一个极其精简的期末真题模型：
假设系统里只有 3 个进程 (P0, P1, P2)，只有 1 种资源。

| 进程 | Max (总共需要) | Allocation (已分配) |
| :--- | :--- | :--- |
| **P0** | 10 | 5 |
| **P1** | 4 | 2 |
| **P2** | 9 | 2 |

**题目已知：目前系统手里还剩下的资源 Available = 3。**
**问题：当前系统安全吗？**

**👉 第一步：自己算出 Need 矩阵 (Need = Max - Allocation)**
*   P0 的 Need = 10 - 5 = **5**
*   P1 的 Need = 4 - 2 = **2**
*   P2 的 Need = 9 - 2 = **7**

**👉 第二步：开始试探，找能满足的人**
*   目前系统 **Available = 3**。
*   看 P0：Need 是 5，但手里只有 3，**不满足**，P0 只能等。
*   看 P1：Need 是 2，手里有 3。**满足！** (3 $\ge$ 2)
*   **让 P1 运行！**
    *   P1 运行完毕后，把它的 Allocation (2) 还给系统。
    *   **新的 Available = 3 + 2 = 5**。
    *   序列里记录下：**[P1]**

**👉 第三步：带着变多的钱，继续找**
*   现在系统 **Available = 5**。
*   看刚才没满足的 P0：Need 是 5。现在手里正好有 5，**满足了！**
*   **让 P0 运行！**
    *   P0 运行完毕后，把它的 Allocation (5) 还给系统。
    *   **新的 Available = 5 + 5 = 10**。
    *   序列更新为：**[P1, P0]**

**👉 第四步：继续**
*   现在系统 **Available = 10**。
*   看最后的 P2：Need 是 7。手里有 10，**满足！**
*   **让 P2 运行！**
    *   P2 运行完毕，归还 Allocation (2)。
    *   **新的 Available = 10 + 2 = 12**。
    *   序列更新为：**[P1, P0, P2]**

**👉 最终结论**：
所有进程都成功运行完毕。**当前系统是安全的，存在一个安全序列：P1 -> P0 -> P2**。
（*注：安全序列可能不止一个，只要你能找出一个完整的，就能拿满分！*）

这个“银行家算法”的核心，其实就是个**简单的加减法游戏**：找一个需求量小于当前库存的进程，让它执行完，然后把它的“已分配”资源加回“库存”里，库存变大了，再去满足下一个进程。

### 🟢 第一部分：彻底讲透“死锁” (Deadlock)

**1. 什么是死锁？（最通俗的本质）**
想象一下：你和室友要吃泡面。你手里拿着“碗”，需要“叉子”；室友手里拿着“叉子”，需要“碗”。
你盯着室友说：“你把叉子给我，我吃完连碗带叉子一起给你。”
室友盯着你说：“你把碗给我，我吃完连碗带叉子一起给你。”
**结果：** 你们俩谁也不肯先放手，互相死等对方，最后双双饿死。这就是**死锁**。

在操作系统里：“你和室友”就是**进程**，“碗和叉子”就是**系统资源（比如打印机、内存）**。

**2. 死锁产生的 4 个必要条件（期末必考简答题/选择题！）**
请注意“必要”二字，意思是：**这 4 个条件必须同时、全部满足，死锁才会发生！只要打破其中任何一个，死锁就解除了。**

*   **① 互斥条件**：资源是独占的。（这个碗一次只能一个人用，不能劈成两半两个人同时吃）。
*   **② 请求和保持条件**：你已经“保持（占有）”了碗，同时又“请求”叉子。而且在拿到叉子之前，你**绝对不放下**手里的碗。
*   **③ 不剥夺条件（不可抢占）**：室友不能一拳把你打晕，强行把你的碗“抢”走。资源只能由占有它的人主动释放。
*   **④ 环路等待条件**：形成了一个闭环。你等室友，室友等你。（A等B，B等C，C等A）。

**3. 银行家算法在死锁里扮演什么角色？**
操作系统的设计者为了对付死锁，提出了几种策略，其中一种叫**“死锁避免”**。
银行家算法就是“死锁避免”的代表作！
它的核心思想是：**每次你想借资源，系统都会像银行长一样，先在脑子里“模拟”推演一遍——如果把资源借给你了，接下来大家会不会陷入“环路等待”的死胡同？如果模拟发现安全，就借给你；如果发现有死锁风险，就让你先等着。**

### 🔴 第二部分：银行家算法 完整期末大题实战

这是国内考试最标准的 3 种资源、5 个进程的题型。请仔细看表格和每一步的状态转化。

#### 📋 题目背景与初始状态
假设系统中有 5 个进程 $P_0 \sim P_4$。
系统有 3 类资源 $A, B, C$。它们的总数量分别是 10, 5, 7。
在 `T0` 时刻，系统的资源分配情况如下表所示：

| 进程名称 | Max (最大需求矩阵) | Allocation (已分配矩阵) | Need (还需要矩阵) | Available (当前可用资源) |
| :--- | :---: | :---: | :---: | :---: |
| | **A \ B \ C** | **A \ B \ C** | **A \ B \ C** | **A \ B \ C** |
| **$P_0$** | 7 \ 5 \ 3 | 0 \ 1 \ 0 | **(自己算)** | **3 \ 3 \ 2** |
| **$P_1$** | 3 \ 2 \ 2 | 2 \ 0 \ 0 | **(自己算)** | |
| **$P_2$** | 9 \ 0 \ 2 | 3 \ 0 \ 2 | **(自己算)** | |
| **$P_3$** | 2 \ 2 \ 2 | 2 \ 1 \ 1 | **(自己算)** | |
| **$P_4$** | 4 \ 3 \ 3 | 0 \ 0 \ 2 | **(自己算)** | |

---

#### ✍️ 第一问：请算出 Need 矩阵的值。
**核心公式：`Need = Max - Allocation`** (每一列的数字对应相减)
*在卷子上，你只需要把计算结果填出来即可：*
*   $P_0$: (7, 5, 3) - (0, 1, 0) = **7, 4, 3**
*   $P_1$: (3, 2, 2) - (2, 0, 0) = **1, 2, 2**
*   $P_2$: (9, 0, 2) - (3, 0, 2) = **6, 0, 0**
*   $P_3$: (2, 2, 2) - (2, 1, 1) = **0, 1, 1**
*   $P_4$: (4, 3, 3) - (0, 0, 2) = **4, 3, 1**

---

#### ✍️ 第二问：请判断 T0 时刻系统是否安全？如果安全，请给出一个安全序列。
**【解题保姆级步骤】（也就是所谓的“安全性算法”）：**

我们需要在草稿纸上找出一个进程的执行顺序，使得每个进程需要的资源都能被当前剩余资源满足。
当前手里拥有的资源总量：**Work = Available = (3, 3, 2)**

**状态转化推导过程：**

1.  **扫描寻找：** 谁的 Need $\le$ Work (3, 3, 2)？
    *   $P_0$ Need(7,4,3) > Work，不行。
    *   $P_1$ Need(1,2,2) $\le$ Work(3,3,2)。**可以！让 $P_1$ 执行。**
2.  **$P_1$ 状态转化：** $P_1$ 拿到资源，顺利干完活。它把原本占用的 Allocation(2,0,0) 全都还给系统！
    *   **新 Work** = 老 Work(3,3,2) + $P_1$归还的(2,0,0) = **(5, 3, 2)**
    *   安全序列记录：**[$P_1$]**
3.  **继续扫描寻找：** 带着新 Work(5, 3, 2)，看看剩下的人谁能满足？
    *   $P_3$ Need(0,1,1) $\le$ Work(5,3,2)。**可以！让 $P_3$ 执行。**
4.  **$P_3$ 状态转化：** $P_3$ 执行完，归还 Allocation(2,1,1)。
    *   **新 Work** = (5,3,2) + (2,1,1) = **(7, 4, 3)**
    *   安全序列记录：**[$P_1, P_3$]**
5.  **继续找：** Work 现在是 (7, 4, 3)。
    *   $P_4$ Need(4,3,1) $\le$ Work(7,4,3)。**可以！让 $P_4$ 执行。**
6.  **$P_4$ 状态转化：** $P_4$ 执行完，归还 Allocation(0,0,2)。
    *   **新 Work** = (7,4,3) + (0,0,2) = **(7, 4, 5)**
    *   安全序列记录：**[$P_1, P_3, P_4$]**
7.  **继续找：** Work 是 (7, 4, 5)。
    *   $P_0$ Need(7,4,3) $\le$ Work(7,4,5)。**可以！让 $P_0$ 执行。**
8.  **$P_0$ 状态转化：** $P_0$ 执行完，归还 Allocation(0,1,0)。
    *   **新 Work** = (7,4,5) + (0,1,0) = **(7, 5, 5)**
    *   安全序列记录：**[$P_1, P_3, P_4, P_0$]**
9.  **最后找：** 只剩 $P_2$。Work 是 (7, 5, 5)。
    *   $P_2$ Need(6,0,0) $\le$ Work(7,5,5)。**可以！让 $P_2$ 执行。**
    *   **最终 Work** = (7,5,5) + $P_2$归还的(3,0,2) = **(10, 5, 7)** *(正好等于最开始系统的所有资源总数，说明算对了！)*
    *   安全序列记录：**[$P_1, P_3, P_4, P_0, P_2$]**

**答题标准话术：**
“因为能找到一个安全序列 $P_1 \rightarrow P_3 \rightarrow P_4 \rightarrow P_0 \rightarrow P_2$ （*注：安全序列不唯一，写出一个即可*），所以 T0 时刻系统是安全的。”

---

#### ✍️ 第三问（压轴）：如果此时 $P_1$ 突然发出资源请求 Request(1, 0, 2)，系统能否把资源分配给它？
这就是银行家算法真正的精髓：“**试探性分配 + 安全性检查**”。

**第一步：合法性与可用性检查（先过两道门槛）**
1.  **查 Need**：请求的量不能超过它自己原来声明的最大需求。
    Request(1,0,2) $\le$ $P_1$的Need(1,2,2)？ **满足。**
2.  **查 Available**：请求的量不能超过银行现在的库存。
    Request(1,0,2) $\le$ 当前Available(3,3,2)？ **满足。**

**第二步：试探性分配（系统在脑海里模拟把资源给它）**
系统假装把资源给了 $P_1$，更新相关数据：
*   **新 Available** = 原 Available(3,3,2) - 分出去的(1,0,2) = **(2, 3, 0)**
*   **$P_1$ 新 Allocation** = 原来已有的(2,0,0) + 新拿到的(1,0,2) = **(3, 0, 2)**
*   **$P_1$ 新 Need** = 原 Need(1,2,2) - 满足了的(1,0,2) = **(0, 2, 0)**

**第三步：安全性算法（拿着新的 Available，重新做一遍第二问的过程）**
现在的初始 Work = **(2, 3, 0)**。
1.  看 $P_0$: Need(7,4,3) > Work(2,3,0)，不行。
2.  看 $P_1$: 新Need(0,2,0) $\le$ Work(2,3,0)。**可以！**
    *   $P_1$执行完归还新Allocation(3,0,2)。
    *   新 Work = (2,3,0) + (3,0,2) = **(5, 3, 2)**
3.  看 $P_3$: Need(0,1,1) $\le$ Work(5,3,2)。**可以！**
    *   $P_3$执行完归还(2,1,1)。
    *   新 Work = (5,3,2) + (2,1,1) = **(7, 4, 3)**
4.  *(后面的过程和第二问完全一样了...)*
    *   $P_4$ 执行，新 Work = (7,4,3) + (0,0,2) = (7,4,5)
    *   $P_0$ 执行，新 Work = (7,4,5) + (0,1,0) = (7,5,5)
    *   $P_2$ 执行，新 Work = (7,5,5) + (3,0,2) = (10,5,7)

**答题标准话术：**
“系统在假设分配后，能找到一个安全序列 $P_1 \rightarrow P_3 \rightarrow P_4 \rightarrow P_0 \rightarrow P_2$。系统处于安全状态，**因此可以同意 $P_1$ 的请求，将资源分配给它。**”

### 💡 核心总结复盘

银行家算法之所以看起来难，是因为数据多，你容易算串行。但它的核心逻辑其实非常死板：
1. **算出每个人还要多少 (Need)**。
2. **拿着银行手里的余钱 (Available)**，去找一个能满足的人。
3. **那个人满足后，会把他手里的钱全部吐出来 (Available = Available + Allocation)**。
4. **银行拿着变多的钱，再去满足下一个人**，直到所有人都满足。

只要你在草稿纸上，用我上面这种 `Work = Work + Allocation` 的方式，一步一步记录下来，这种题就是白送的满分。

你看这次的排版和状态推导过程是否清晰？试着自己拿一张草稿纸，把第三问里我写的“试探性分配”的加减法自己算一遍，如果能顺滑地算到 `(10,5,7)`，那期末这道压轴题你就彻底拿下了！

### 第一问＆第二问：判断 T0 时刻系统是否安全

**初始已知条件（T0时刻）：**
系统初始手里拥有的资源 `Available = (3, 3, 2)`。
*（注意：表格里的 `Work` 就代表系统当前手里还剩的可用资源，也就是拿着去不断投资的“本金”）*

**✅ T0 时刻安全性检查表（推导过程表格化）：**

| 查找顺序 | 进程 | Work (系统当前可用) <br> `A  B  C` | Need (还需要) <br> `A  B  C` | Allocation (已分配) <br> `A  B  C` | Work + Allocation (执行完后系统总可用) <br> `A  B  C` | Finish <br> (是否成功) |
| :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| **步骤 1** | **$P_1$** | **3 \ 3 \ 2** | 1 \ 2 \ 2 | 2 \ 0 \ 0 | 3+2 \ 3+0 \ 2+0 = **5 \ 3 \ 2** | True |
| **步骤 2** | **$P_3$** | **5 \ 3 \ 2** | 0 \ 1 \ 1 | 2 \ 1 \ 1 | 5+2 \ 3+1 \ 2+1 = **7 \ 4 \ 3** | True |
| **步骤 3** | **$P_4$** | **7 \ 4 \ 3** | 4 \ 3 \ 1 | 0 \ 0 \ 2 | 7+0 \ 4+0 \ 3+2 = **7 \ 4 \ 5** | True |
| **步骤 4** | **$P_0$** | **7 \ 4 \ 5** | 7 \ 4 \ 3 | 0 \ 1 \ 0 | 7+0 \ 4+1 \ 5+0 = **7 \ 5 \ 5** | True |
| **步骤 5** | **$P_2$** | **7 \ 5 \ 5** | 6 \ 0 \ 0 | 3 \ 0 \ 2 | 7+3 \ 5+0 \ 5+2 = **10 \ 5 \ 7** | True |

**表格看法指南（以步骤1为例）：**
1. 拿着本金 `Work (3,3,2)`，发现 $P_1$ 的要求 `Need (1,2,2)` 可以满足。
2. 让 $P_1$ 执行完。它把以前借的 `Allocation (2,0,0)` 还给系统。
3. 系统的本金变大了，变成了 `Work + Allocation = (5,3,2)`。
4. 拿着变大的本金 `(5,3,2)` 进入步骤2，继续找下一个能满足的人...

**结论：** 所有进程的 Finish 都变为了 True，存在安全序列 $\{P_1, P_3, P_4, P_0, P_2\}$，系统安全。

### 第三问：$P_1$ 请求资源 Request(1, 0, 2)，能否分配？

**第一步：检查请求是否合法（极简记录）**
1. `Request(1,0,2) ≤ P1_Need(1,2,2)`，合法。
2. `Request(1,0,2) ≤ Available(3,3,2)`，系统有钱，可以借。

**第二步：系统假装分配，更新 $P_1$ 和 Available 的数据**
*   **系统剩余 (Available)** = `(3,3,2) - (1,0,2)` = **(2, 3, 0)**  *← 这是新的初始本金*
*   **$P_1$ 还需要 (Need)** = `(1,2,2) - (1,0,2)` = **(0, 2, 0)**
*   **$P_1$ 已分配 (Allocation)** = `(2,0,0) + (1,0,2)` = **(3, 0, 2)**

*(注意：此时其他进程的数据完全没变，只有 $P_1$ 和 系统的钱变了)*

**第三步：用新的数据，再画一次“安全性检查表”**

**✅ 分配给 $P_1$ 后的安全性检查表：**

| 查找顺序 | 进程 | Work (系统当前可用) <br> `A  B  C` | Need (还需要) <br> `A  B  C` | Allocation (已分配) *(注意P1更新了)* <br> `A  B  C` | Work + Allocation (执行完后系统总可用) <br> `A  B  C` | Finish <br> (是否成功) |
| :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| **步骤 1** | **$P_1$** | **2 \ 3 \ 0** | **0 \ 2 \ 0** | **3 \ 0 \ 2** | 2+3 \ 3+0 \ 0+2 = **5 \ 3 \ 2** | True |
| **步骤 2** | **$P_3$** | **5 \ 3 \ 2** | 0 \ 1 \ 1 | 2 \ 1 \ 1 | 5+2 \ 3+1 \ 2+1 = **7 \ 4 \ 3** | True |
| **步骤 3** | **$P_4$** | **7 \ 4 \ 3** | 4 \ 3 \ 1 | 0 \ 0 \ 2 | 7+0 \ 4+0 \ 3+2 = **7 \ 4 \ 5** | True |
| **步骤 4** | **$P_0$** | **7 \ 4 \ 5** | 7 \ 4 \ 3 | 0 \ 1 \ 0 | 7+0 \ 4+1 \ 5+0 = **7 \ 5 \ 5** | True |
| **步骤 5** | **$P_2$** | **7 \ 5 \ 5** | 6 \ 0 \ 0 | 3 \ 0 \ 2 | 7+3 \ 5+0 \ 5+2 = **10 \ 5 \ 7** | True |

*(如果你仔细看，会发现从步骤2开始，后面的过程和第一问一模一样，因为 $P_1$ 还给系统的总金额和以前是一样的！)*

**结论：** 假装分配后，依然能找到安全序列 $\{P_1, P_3, P_4, P_0, P_2\}$。说明假装分配是安全的，**系统决定正式把资源借给 $P_1$**。

### 🛡️ 尾声知识点 1：死锁的预防 (Deadlock Prevention)

**【考试重点】**
前面我们背过死锁的 4 个必要条件（互斥、请求和保持、不剥夺、环路等待）。
**死锁预防的核心思想就是：只要我破坏掉其中任意一个条件，死锁就永远不可能发生！**

考试常考怎么破坏，以及这些方法的缺点（通常是多选题或判断题）：

1.  **破坏“互斥”条件**
    *   *做法*：把只能独占的设备，改造成大家能共享的设备。
    *   *常考技术名词*：**SPOOLing 技术**（比如把一台打印机虚拟成多台，大家把打印任务放进队列里排队）。
    *   *缺点*：很多设备（如大部分硬件）根本没法共享，所以这个方法局限性很大。
2.  **破坏“请求和保持”条件**
    *   *做法*：**静态分配法**。进程在运行前，必须**一次性**把所有需要的资源全申请到手。只要差一个，你就一个都别拿，乖乖等着。
    *   *缺点*：**极其浪费资源**（比如你最后一天才需要打印机，但你从第一天就把它占着），还会导致别的进程“饿死”。
3.  **破坏“不剥夺”条件**
    *   *做法*：进程手里拿着一些资源，如果去申请新资源被拒绝了，它必须**立刻主动释放**手里已经拿着的所有资源。以后再重新申请。
    *   *缺点*：反复借了还、还了借，开销太大，会导致前功尽弃（常用于CPU和内存，不适用于打印机）。
4.  **破坏“环路等待”条件 🔥（最爱考）**
    *   *做法*：**顺序资源分配法**。给系统里所有的资源编号（比如：1号鼠标、2号键盘、3号打印机）。规定所有进程必须**严格按照编号递增的顺序**去申请资源。
        *   *(大白话：如果你手里已经拿了3号，你就绝不能回头去申请1号和2号，只能去申请4号。这就永远不可能形成闭环！)*
    *   *缺点*：编号固定，如果新加了设备很不方便；进程实际使用资源的顺序可能和编号不一样，导致浪费。

### 🩺 尾声知识点 2：死锁的检测与解除 (Detection & Recovery)

**【考试重点】**
系统不去预防，也不去用银行家算法避免，而是**放任死锁发生**。系统隔一段时间去“体检”（检测），如果发现死锁了，就强行“做手术”（解除）。

**1. 怎么检测？（常考选择题）**
*   **资源分配图**：操作系统里用圆圈代表“进程”，用方框代表“资源”。画出谁占用了谁，谁在等谁的箭头图。
*   **死锁定理（核心考点）**：如果这个资源分配图能够被**完全简化**（也就是所有能拿到资源的进程都能顺利跑完并释放资源，最后图里一个箭头都不剩），就没有死锁；如果**化简不下去**，剩下一个死结，就是死锁。

**2. 怎么解除？（常考简答/填空）**
发现死锁后，系统怎么“痛下杀手”？主要有 3 招：
1.  **资源剥夺法**：挂起某些死锁进程，强行抢走它的资源分给别人。（相当于警察来了，强行把你室友手里的叉子抢过来给你吃）。
2.  **撤销进程法（终止进程法）**：最粗暴。直接把参与死锁的进程全部杀掉（Kill），或者按照优先级一个一个杀，直到死锁解开。
3.  **进程回退法**：让进程退回到死锁发生前的一个“存档点”（检查点），重新来过。
