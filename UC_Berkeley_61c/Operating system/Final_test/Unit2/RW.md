### 第一部分：大纲核心大题 —— 读者-写者问题

**1. 题目场景与规则设定：**
系统中有多个并发进程共享一个数据区（如文件或数据库）。
*   **读者进程**：只要求读数据，不修改。**允许多个读者同时读**。
*   **写者进程**：要求修改数据。**写者执行时，绝对排他**（即不允许其他写者写，也不允许任何读者读）。

**2. 核心技术难点：**
写者好处理，直接用一个互斥信号量锁住即可。难点在于**读者**：
如果用普通的 P/V 操作，第一个读者进去上锁，第二个读者就会被挡在外面，无法实现“允许多个读者同时读”。

**3. 解决方案：引入“计数器”机制（首锁尾解）**
*   **核心思想**：我们引入一个整型变量 `Rcount` 来记录当前正在读的读者数量。
    *   当 `Rcount == 0` 时，说明当前是**第一个读者**准备进入，由他负责**对数据区加锁**（挡住写者）。
    *   中间进入的读者，只增加 `Rcount` 的值，不操作数据区的锁。
    *   当 `Rcount` 减到 `0` 时，说明当前是**最后一个退出**的读者，由他负责**对数据区解锁**（放行写者）。

**4. 需要的变量与信号量：**
*   `int Rcount = 0;` ：记录当前读者数量。
*   `semaphore Wmutex = 1;` ：**数据区互斥锁**。用于实现“读-写”互斥、“写-写”互斥。
*   `semaphore Rmutex = 1;` ：**计数器互斥锁**。因为 `Rcount` 是所有读者共享的变量，为防止并发修改出错，必须对其进行互斥保护。

**5. 标准伪代码（考试常考填空或默写）：**

```c
int Rcount = 0;
semaphore Wmutex = 1;
semaphore Rmutex = 1;

// ================= 读者进程 =================
void Reader() {
    while(true) {
        // --- 准备进入读区 ---
        P(Rmutex);              // 对 Rcount 变量加锁
        if (Rcount == 0) {
            P(Wmutex);          // 【考点】如果是第一个读者，负责锁住数据区，阻止写者进入
        }
        Rcount++;               // 读者数量加 1
        V(Rmutex);              // 对 Rcount 变量解锁

        // --- 执行读操作 ---
        read_data();            // 此时可以有多个读者并发执行到这一行

        // --- 准备离开读区 ---
        P(Rmutex);              // 对 Rcount 变量加锁
        Rcount--;               // 读者数量减 1
        if (Rcount == 0) {
            V(Wmutex);          // 【考点】如果是最后一个读者，负责解开数据区的锁，唤醒写者
        }
        V(Rmutex);              // 对 Rcount 变量解锁
    }
}

// ================= 写者进程 =================
void Writer() {
    while(true) {
        P(Wmutex);              // 申请数据区独占锁。如果有读者在读，或有其他写者在写，这里会阻塞
        write_data();           // 执行写操作
        V(Wmutex);              // 释放数据区独占锁
    }
}
```

#### 补充考点 1：大纲 2.5 节括号中的“信号量机制（记录型/整型）”

操作系统发展过程中，信号量分为两种，考试常考它们的区别：

*   **整型信号量**：
    *   **定义**：只包含一个代表资源数量的整数。
    *   **底层逻辑**：当资源不够时，代码会进入一个 `while(S <= 0);` 的空循环，不断测试资源是否可用。
    *   **致命缺陷**：导致进程在 CPU 上死循环等待，**违反了临界区的“让权等待”原则**（即进不去临界区时，应立刻释放 CPU）。这被称为“忙等”（Busy Waiting）。
*   **记录型信号量（考试重点代码默认使用的都是这个）**：
    *   **定义**：不仅包含资源数量的整数 `value`，还包含一个**进程等待队列（阻塞队列）**链表指针。
    *   **底层逻辑**：当资源不够（`value < 0`）时，系统会调用 `block()` 原语，将当前进程放入等待队列，并切换 CPU 去执行别的进程。
    *   **优势**：完美解决了忙等问题，**遵循了“让权等待”原则**。

#### 补充考点 2：大纲 2.3 节中的“进程切换”

大纲里“进程控制”提到了：创建、终止、阻塞、唤醒、**切换**。前面详细讲了前四个，这里补齐“切换”。

*   **进程切换的定义**：CPU 从一个正在运行的进程上剥夺执行权，转而去执行另一个就绪进程的过程。
*   **切换到底切换了什么？（简答题考点）**
    1.  保存当前进程的**处理机上下文**（把 CPU 的程序计数器 PC、通用寄存器等数据，保存到当前进程的 PCB 中）。
    2.  更新当前进程 PCB 的状态（如从运行态改为就绪态或阻塞态），并移入相应队列。
    3.  调度另一个进程，从其 PCB 中恢复上下文（把之前保存的寄存器数据写回 CPU）。
*   **注意与“模式切换”（用户态到内核态）区分**：模式切换不一定改变当前运行的进程，而进程切换必然伴随着模式切换（必须进入内核态才能执行切换原语）。

### 2. 逐行死磕：读者进程（Reader）

读者进场和退场的时候，代码是对称的，核心就是实现“第一个读者进来上锁，最后一个读者走了解锁”。

```c
void Reader() {
    while(true) {
        // ------ 【第一步：进场登记】 ------
        P(Rmutex);              // 1. Rmutex 减 1（从 1 变 0），读者 A 反锁登记处的门，不让其他读者进来干扰
        if (Rcount == 0) {      
            P(Wmutex);          // 2. 【核心卡点】：如果 Rcount 是 0，说明 A 是第一个来的读者！
                                //    它立刻执行 P(Wmutex)，把 Wmutex 减 1（从 1 变 0），把数据区的大门锁死！
                                //    这样，外面的写者就绝对进不来了。
        }
        Rcount++;               // 3. 登记：屋里的读者人数加 1（Rcount 从 0 变成 1）
        V(Rmutex);              // 4. Rmutex 加 1（从 0 变回 1），解开登记处的门，让后面的读者 B 进来登记

        // ------ 【第二步：快乐读书】 ------
        read_data();            // 5. 正在读数据。注意：此时后面的读者 B 进来登记完后，
                                //    因为 Rcount 已经不是 0 了，B 会跳过第 2 步，直接进来和 A 一起读！

        // ------ 【第三步：退场注销】 ------
        P(Rmutex);              // 6. Rmutex 减 1（锁住登记处），准备悄悄离开
        Rcount--;               // 7. 注销：走了一个人，读者人数减 1
        if (Rcount == 0) {      
            V(Wmutex);          // 8. 【核心卡点】：如果减完后 Rcount 变成 0 了，说明他是最后离开的那个人！
                                //    他负责执行 V(Wmutex)，把 Wmutex 加 1（从 0 变回 1），把数据区的大门打开。
                                //    挂在外面睡觉的写者此时就会被系统叫醒，终于可以进来写了。
        }
        V(Rmutex);              // 9. Rmutex 加 1（解开登记处的门），退场完毕
    }
}

```

## 1. 核心变量与信号量的底层定义

在看代码前，必须先在内存和操作系统的角度看清楚这三个变量到底是什么：

```c
int Rcount = 0;       

```

* **物理本质**：这是一个普通的**全局整型变量**（位于进程的堆/全局数据区）。
* **作用**：单纯用于记录当前正处于临界区（读区）内的读者进程总数。

```c
semaphore Rmutex = 1; 
semaphore Wmutex = 1; 

```

* **物理本质**：它们是操作系统内核管理的**记录型信号量（Structure）**。在底层，它们不是简单的整数，而是一个包含两个成员的结构体：

$$\text{semaphore} = \{ \text{int value}; \ \text{struct PCB *queue} \}$$


* `value`：代表可用资源的数量（初值为 1，表示资源可用）。
* `queue`：代表一个进程等待队列（阻塞队列）的链表指针。

## 2. 逐行死磕：读者进程（Reader）的底层执行逻辑

```c
void Reader() {
    while(true) {
        // ------------------ 【进场控制区】 ------------------
        P(Rmutex);              

```

* **底层原理**：这是对信号量 `Rmutex` 执行 P 操作。
1. 内核将 `Rmutex.value` 自减 1（值从 1 变成 0）。
2. 判断结果：因为 $0 \ge 0$，说明没有冲突，当前读者进程不需要阻塞，直接向下执行。


* **作用**：**保护全局变量 `Rcount` 的原子性**。只要这个进程把 `Rmutex` 减成了 0，后续其他读者进程如果执行到这一行，自减后值会变成 -1（$<0$），就会被操作系统强行剥夺 CPU，挂起到 `Rmutex.queue` 队列中睡觉。这保证了同一时刻，**只有一个读者能修改 `Rcount**`。

```c
        if (Rcount == 0) {
            P(Wmutex);          
        }

```

* **底层原理**：
1. 判断当前全局变量 `Rcount` 是否等于 0。如果等于 0，说明当前进程是**第一个试图进入读临界区的读者**。
2. 既然是第一个，它必须代表整个读者群体去申请数据区的访问权。于是对 `Wmutex` 执行 P 操作：内核将 `Wmutex.value` 自减 1。
3. **若此时没有写者在执行**：`Wmutex.value` 从 1 变 0（$\ge 0$），该读者成功通过。
4. **若此时正有写者在执行**：`Wmutex.value` 之前已经被写者改成了 0，读者执行自减后变成 -1（$<0$）。**该读者进程立刻被内核阻塞，挂起到 `Wmutex.queue` 队列中。**（注意：此时该读者还握着 `Rmutex` 的锁，所以后续的所有读者都会被卡在第一步 `P(Rmutex)` 处，全部阻塞）。


* **作用**：**实现“读-写”互斥**。第一个读者负责把写者挡在外面。

```c
        Rcount++;               

```

* **底层原理**：在 CPU 寄存器中将全局变量 `Rcount` 的值加 1。
* **作用**：记录现在多了一个读者正在临界区内。

```c
        V(Rmutex);              

```

* **底层原理**：这是对信号量 `Rmutex` 执行 V 操作。
1. 内核将 `Rmutex.value` 自加 1（值从 0 变回 1）。
2. 判断结果：因为 $1 > 0$（说明之前没有其他读者因为抢这把锁而阻塞在队列里），进程继续向下。


* **作用**：**释放对 `Rcount` 的保护锁**。允许下一个读者进程进来修改 `Rcount`。

```c
        // ------------------ 【临界区（读区）】 ------------------
        read_data();            

```

* **底层原理**：执行实际的读取内存/文件操作。
* **作用**：在此处，因为 `Rmutex` 已经释放，后续的读者进程可以并发地通过上面的逻辑，一同执行到这一行。**实现了多个读者同时读取的并发要求。**

```c
        // ------------------ 【退场控制区】 ------------------
        P(Rmutex);              

```

* **底层原理**：读者读完准备离开了，再次对 `Rmutex` 执行 P 操作（自减 1，值从 1 变 0）。
* **作用**：再次锁住全局变量 `Rcount`，防止多个退场的读者同时修改人数导致计数错乱。

```c
        Rcount--;               

```

* **底层原理**：将全局变量 `Rcount` 的值减 1。
* **作用**：注销退出，读者人数减 1。

```c
        if (Rcount == 0) {
            V(Wmutex);          
        }

```

* **底层原理**：
1. 判断减 1 后的 `Rcount` 是否等于 0。如果等于 0，说明当前进程是整个读临界区里**最后一个离开的读者**。
2. 既然没有人再读了，它必须代表读者群体释放数据区的控制权。于是对 `Wmutex` 执行 V 操作：内核将 `Wmutex.value` 自加 1。
3. **若此时外面有写者在等待**：在加 1 之前，`Wmutex.value` 必然是负数（因为写者被卡住了），加 1 后结果仍 $\le 0$。内核检测到这个状态，**立刻从 `Wmutex.queue` 队列中唤醒第一个被阻塞的写者进程**，将其移入就绪队列，等待 CPU 调度。


* **作用**：**实现“读-写”恢复**。最后一个离开的读者负责开门并唤醒写者。

```c
        V(Rmutex);              
    }
}

```

* **底层原理**：将 `Rmutex.value` 自加 1（变回 1）。
* **作用**：释放锁，退场完毕。

## 3. 逐行死磕：写者进程（Writer）的底层执行逻辑

```c
void Writer() {
    while(true) {
        P(Wmutex);              

```

* **底层原理**：写者准备写入数据，对核心信号量 `Wmutex` 执行 P 操作：
1. 内核将 `Wmutex.value` 自减 1。
2. **情况 A（数据区空闲）**：原值为 1，减完变 0（$\ge 0$），写者成功通过，进入临界区。
3. **情况 B（当前有读者在读，或有其他写者在写）**：原值已经是 0（或负数），自减后变成负数（$<0$）。**内核立刻剥夺当前写者的 CPU 执行权，将其强行挂入 `Wmutex.queue` 阻塞队列中挂起。**


* **作用**：**实现“写-写”互斥与“读-写”互斥**。只要里面有人，写者绝对进不去。

```c
        write_data();           

```

* **底层原理**：执行写内存/修改文件的硬件指令。
* **作用**：此时整个系统只有这一个写者能操作该数据区，保证了数据的完整性。

```c
        V(Wmutex);              
    }
}

```

* **底层原理**：写者执行完毕，离开临界区，对 `Wmutex` 执行 V 操作：
1. 内核将 `Wmutex.value` 自加 1。
2. 判断结果：如果加 1 后结果仍 $\le 0$，说明 `Wmutex.queue` 队列里还有人在排队（可能是其他写者，也可能是第一个被卡住的读者）。**内核会调用唤醒原语（Wakeup），将队列头部的第一个进程唤醒**。


* **作用**：**释放独占权，并唤醒后续排队进程**。

## 4. 终极逻辑闭环

请注意这两个**绝对互斥**的底层状态：

* 当 `Wmutex.value == 0` 时，代表**要么有一个写者在临界区内，要么有至少一个读者在临界区内**。
* 当 `Wmutex.value < 0` 时，其**负数的绝对值**（如 -2），精确代表**当前在外面长椅（`Wmutex.queue`）上被阻塞挂起的进程数量**。
