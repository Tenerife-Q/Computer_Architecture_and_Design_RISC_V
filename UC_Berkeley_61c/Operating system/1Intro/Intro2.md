太好了，我们继续。第一阶段我们把程序放进了内存，并把 CPU 交给了它。但这个时候，程序处于一个被“严格限制”的状态。

接下来的第二阶段，我们将深入探讨**控制单元（Control Unit）**和**状态寄存器（Status Registers）**是如何在数字逻辑层面实现“安全隔离”的，以及应用程序如何通过**系统调用（Syscall）**与底层硬件打交道。（对应图2、图3、图6、图7）

### 第三阶段：硬件级别的权限隔离 (Protection and Privilege - 图3, 图6)

如果任由应用程序直接执行所有 CPU 指令，一旦程序存在除零错误、内存越界或者包含恶意代码，整个计算机系统就会立刻崩溃。因此，**操作系统（OS）的权威必须由 CPU 硬件本身来强制保障**。

#### 1. 状态寄存器与执行模式 (Execution Modes)
在 RISC-V 架构中，CPU 内部包含一组特殊的寄存器，称为 **控制与状态寄存器 (CSRs, Control and Status Registers)**。其中有一个极其关键的硬件位（Bit），我们称之为 **特权模式位 (Privilege Mode Bit)**。

*   **用户模式 (User Mode, U-Mode)**：当特权模式位 = `0` 时。
    *   **硬件限制**：在流水线的“指令译码 (ID)”阶段，如果控制单元发现当前是 U-Mode，且取到的指令是一条特权指令（例如修改内存映射表、直接读写 I/O 端口、关闭中断），硬件会直接**拒绝执行**，并抛出一个“非法指令异常”。
*   **监管者模式 (Supervisor Mode, S-Mode)**：当特权模式位 = `1` 时。
    *   **最高权限**：OS 运行在此模式。CPU 允许执行所有指令，可以修改 CSRs，可以访问所有物理内存空间。
*   *(注：图3中提到 RISC-V 还有一个底层的 Machine Mode，那是比 OS 更底层的硬件固件级权限，通常 OS 运行在 S-Mode 即可)*。

**核心逻辑**：应用程序**绝对无法**通过一条普通的汇编指令（比如 `addi` 或 `sw`）直接把这个模式位从 `0` 改成 `1`。想要跨越这个权限边界，必须依赖 CPU 提供的一种极为特殊的机制——**陷入 (Trap/Interrupt)**。

### 第四阶段：系统调用 (Syscalls) 的数字逻辑全过程 (图7)

既然应用程序被关在了用户模式的“沙箱”里，当它需要读取磁盘文件（比如运行 `printf` 或读档）、或者申请更多内存（`malloc`）时，该怎么办？它必须向 OS 发起**系统调用 (Syscall)**。

这并非一次简单的函数调用（普通的 `jal` 指令），而是一次**跨越特权级的强制上下文切换**。我们来看看在 RISC-V 汇编层面，一次读取文件的 Syscall 是如何完成的：

#### 1. 准备参数 (Setup Arguments)
应用程序依然在 U-Mode 下运行。编译器会将参数放入标准寄存器中：
*   例如，将文件描述符放入 `a0` 寄存器。
*   将准备接收数据的内存地址放入 `a1` 寄存器。
*   最重要的一步：将**系统调用号 (Syscall ID)** 放入 `a7` 寄存器（比如 Linux 中，读文件的 syscall 编号可能是 63）。

#### 2. 触发软件中断 (Raise Software Interrupt - `ecall`)
参数准备好后，应用程序执行一条极为特殊的汇编指令：**`ecall` (Environment Call)**。
这是引发“同步异常”的指令，也是进入 OS 的唯一合法正门。

#### 3. 硬件的原子操作 (Hardware Trap Mechanism - 极度关键)
当 CPU 的译码单元读取到 `ecall` 指令时，**硬件电路**会在一个极短的时钟周期内自动完成以下“原子操作”（不可被打断）：
1.  **保存断点**：将当前的程序计数器（PC 指针）的值，强行写入一个叫 `sepc` (Supervisor Exception PC) 的特殊寄存器中，记住是从哪被打断的。
2.  **权限升级**：将状态寄存器中的“特权模式位”强制从 `0` 翻转为 `1` (切换为 S-Mode)。
3.  **强制跳转**：将 PC 指针强制替换为 `stvec` (Supervisor Trap Vector) 寄存器里的值。这个 `stvec` 里面存的正是 **操作系统内核中 Trap Handler（异常处理程序）的物理入口地址**。

#### 4. OS 接管并处理
此刻，CPU 已经在运行 OS 的代码，并且拥有最高权限。
*   OS 检查 `a7` 寄存器，发现里面是 63，明白了：“哦，你要读文件”。
*   OS 检查 `a0` 和 `a1` 里的参数，确保你没有试图读取系统的密码文件（权限校验）。
*   OS 操控硬盘控制器，把数据读取到你指定的内存 `a1` 处。

#### 5. 安全返回 (`sret`)
OS 办完事后，执行另一条特权指令：**`sret` (Supervisor Return)**。
此时硬件电路再次执行原子反向操作：
1.  将特权模式位从 `1` 降级回 `0` (回到 U-Mode)。
2.  将 PC 的值从 `sepc` 中取回。

CPU 回到了你的应用程序中紧接着 `ecall` 的下一条指令继续执行。从应用程序的视角来看，它只是执行了一个指令，然后奇迹般地，目标内存里就填满了硬盘上的数据。

### 第五阶段：宏观总结：大堂经理的真正工作 (What Does the OS do? - 图2)

通过上面这套 **特权模式 + ecall 陷入机制**，操作系统掌握了对硬件的绝对控制权。基于此，图2 总结了 OS 能够实现的三大宏观功能：

1.  **管理底层硬件 (Finds and controls all devices)**：因为普通应用无法直接操作硬件寄存器，只能通过 Syscall，所以 OS 可以统一调配打印机、网卡、磁盘，防止冲突。
2.  **提供高级抽象服务 (Starts services)**：普通程序不需要知道硬盘是怎么转的（物理扇区、磁道），OS 提供了“文件系统”这个抽象概念，程序只需知道“文件路径”。
3.  **多路复用与隔离 (Time-sharing & Isolation)**：
    *   **复用**：因为 OS 能够随时通过硬件定时器产生的中断（Timer Interrupt）强行剥夺 CPU 的控制权，所以它能让多个程序分时运行。
    *   **隔离**：这是接下来的重点（图3提到的 **Memory Translation**），OS 配合硬件 MMU（内存管理单元），为每个进程生成一套独立的虚拟地址映射表。程序 A 如果想访问程序 B 的地址，在硬件地址翻译阶段就会直接报错，并触发陷入，OS 接管后会直接把违规的程序 A 强制杀掉（这就是我们常见的 Segmentation Fault 段错误）。

**（第二步总结）**

至此，这 7 张幻灯片的核心技术逻辑已经全盘托出：
*   **启动流**：ROM 引导 -> 磁盘 Bootloader -> DRAM 内核运行。
*   **权限流**：状态寄存器区分 U-Mode / S-Mode。
*   **控制流**：程序准备寄存器 -> 执行 `ecall` -> 硬件提升权限并跳转 PC -> OS 查表处理 -> `sret` 降权返回。
