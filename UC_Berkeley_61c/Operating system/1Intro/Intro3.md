### 第六阶段：控制流的突变 —— Trap 机制与分类 (图1, 图4, 图5)

在正常的程序执行中，控制流是线性的（PC = PC + 4）或者是基于分支指令跳转的。但系统运行中总会出现必须打断当前程序的事件，这种打断并把控制权交给 OS 的动作，统称为 **陷入 (Trap)**。

在 CS61C 的术语中，Trap 包含两个子类，这**极其关键**，是数字逻辑层面处理方式不同的根本原因：

#### 1. 中断 (Interrupt) —— 外部的、异步的
*   **触发源**：来自 CPU 核心外部的硬件设备。例如：键盘按下发送的电平信号、磁盘控制器完成数据读取、或者主板上的硬件定时器（Timer）倒计时清零。
*   **异步性 (Asynchronous)**：中断与 CPU 当前正在执行的指令**毫无关系**。
*   **处理时机**：因为是异步的，CPU 不需要立刻在当前时钟周期响应。它可以等当前指令稳妥地执行完毕后，在“方便的时候”再跳转去处理中断（"Whenever it's convenient"）。

#### 2. 异常 (Exception) —— 内部的、同步的
*   **触发源**：由 CPU **正在执行的当前指令**本身引起的。
*   **分类**：
    *   **非故意异常**：例如除以零（ALU 报错）、访问了没有权限的内存地址（MMU 报错）、读取到了非法的机器码（译码器报错）。
    *   **故意异常**：例如我们上一节提到的 `ecall`（向 OS 请求系统调用）或 `ebreak`（触发断点调试）。
*   **同步性 (Synchronous)**：异常极其紧急。CPU 必须在**引发异常的那条指令上**精准地停下来，"Drop whatever you are doing and act now"（立刻放下手头的事去处理）。

### 第七阶段：硬件设计危机 —— 流水线中的精确异常 (图2, 图3)

在单周期 CPU 里，处理异常很简单，停下来就行。但是，在我们在之前课程学过的 **5级流水线（IF-ID-EX-MA-WB）**中，同一时刻有 5 条指令在飞，这就引出了巨大的硬件设计难题。

#### 1. 异常可能在任何阶段爆发 (图2)
*   **IF 阶段 (取指)**：PC 地址可能根本不存在，引发 **PC address exception**。
*   **ID 阶段 (译码)**：取出的 32 位机器码查无此指令，引发 **Illegal opcode**。
*   **EX 阶段 (执行)**：ALU 算术溢出，或除以零。
*   **MA 阶段 (访存)**：试图向只读内存（.text段）写入数据，引发 **Data address exception**。

#### 2. 硬件强制要求：精确陷入 (Precise Traps - 图3)
如果 MA 阶段的一条指令触发了异常，此时 EX、ID、IF 阶段还有 3 条更晚进入流水线的指令正在执行。硬件该怎么办？
现代计算机架构强制要求必须实现 **精确陷入 (Precise Traps)**。对 Trap Handler (OS) 来说，当它接管 CPU 时，必须看到一个极其干净的现场：
1.  **之前**：触发异常的指令**之前**的所有指令，必须 100% 执行完毕（结果已写入寄存器或内存）。
2.  **之后**：触发异常的指令**及其之后**的所有指令，必须像**从未执行过一样**（绝对不能修改任何寄存器或内存）。

#### 3. 硬件的数字逻辑实现 (How to handle it)
为了实现精确陷入，硬件将异常视为一种**极端的数据/控制冒险 (Pipeline Hazards)**。当异常在流水线某一级被检测到时，控制单元 (Control Unit) 会执行以下硬核操作：
1.  **冲刷流水线 (Flush)**：将引发异常的指令，以及它后面已经进入流水线的指令，全部转换为空指令 `NOP` (即全0，也就是 `addi x0, x0, 0`)。这确保了它们不会在后续的 WB 阶段修改机器状态。
2.  **记录状态**：硬件自动将异常代码写入 `scause` 寄存器，将引发异常的 PC 地址写入 `sepc` 寄存器。
3.  **强制跳转**：将 PC 强制覆盖为 Trap Handler 的入口地址。

当 OS 处理完异常（例如给程序分配了新的内存页）后，执行 `sret` 指令，CPU 会将 `sepc` 的值送回 PC，**重新取指并执行那条刚才被冲刷掉的指令**。

*(图3 底部提到，在超标量乱序执行处理器 out-of-order processor 中实现精确陷入是极其复杂的硬件工程，但这是保证软件能正常运行的前提。)*

### 第八阶段：时空分割 —— 上下文切换与虚拟内存的伏笔 (图6, 图7)

有了上面精确的 Trap 机制，操作系统终于有了强大的武器来管理进程了。

#### 1. 上下文切换的底层执行步骤 (Context Switch - 图6)
单核 CPU 上同时跑多个程序（多道程序设计 Multiprogramming），依靠的是**硬件定时器中断 (Timer Interrupt)** 和**调度 (Scheduling)**。

假设 CPU 正在执行进程 A（PC指向A的代码）：
1.  **定时器触发**：倒计时 10ms 结束，硬件向 CPU 发出**外部中断**信号。
2.  **陷入 OS**：流水线完成当前指令，将后续指令转为 NOP，提权进入 S-Mode，跳转到 OS 的 Handler。
3.  **保存现场 (Store Process State)**：OS 会在内存中专门为进程 A 开辟一块结构（PCB，进程控制块），将 CPU 当前所有的 32 个通用寄存器的值、以及 `sepc`（进程A被中断的PC位置）**全部存储（Store/sw 指令）到这块内存中**。
4.  **调度与恢复现场 (Load Process State)**：OS 决定下一个运行进程 B。OS 从内存中找到进程 B 之前保存的 PCB，将其中的 32 个寄存器值**全部加载（Load/lw 指令）回 CPU 的物理寄存器中**。
5.  **切换与执行**：OS 设定好新的定时器，通过 `sret` 降权回到 U-Mode，此时 PC 指针被恢复成了进程 B 之前的地址。进程 B 开始执行，对它而言，它根本不知道自己曾经被停下来过。

#### 2. 致命冲突与虚拟内存的需求 (图7)
上下文切换解决了 CPU 时间的分配，但引出了**物理内存的致命冲突**。
*   **冲突现象**：如果进程 A 和进程 B 在编译时，都被设定为从物理地址 `0x8FFFFFFF` 开始运行（这在常规编译器中很常见）。当 OS 切换到 B 时，B 去读取 `0x8FFFFFFF` 的数据，读到的却是 A 留在那里的一半数据！或者更糟，A 的恶意代码故意去覆盖 B 的物理内存地址。
*   **仅仅有特权模式是不够的**：Supervisor mode 只能防止普通程序执行特权指令，但无法阻止普通程序发起一条合法的 `sw t0, 0(s1)` 指令去恶意覆写内存。
*   **终极解决方案 —— 虚拟内存 (Virtual Memory)**：
    这也是接下来几节课的核心内容。OS 和硬件配合，通过分页机制 (Paging)，给每个进程提供一种**假象**：
    每个进程都以为自己独占一个从 `0x00000000` 到 `0xFFFFFFFF` 的完整地址空间。当进程发出读写某个“虚拟地址”的指令时，CPU 内部的 **MMU (内存管理单元)** 会在微秒级别查表，将其动态**翻译 (Translation)** 成真实 DRAM 里的物理地址。

通过这种映射，进程 A 的 `0x8FFFFFFF` 实际上被映射到了物理内存的 `0x1000`，而进程 B 的 `0x8FFFFFFF` 被映射到了物理内存的 `0x4000`。它们在物理层面上被彻底隔离了。

---

**总结全篇架构**：
*   **开机**：通过 ROM Boot -> 磁盘，把 OS 加载进 DRAM。
*   **防御机制**：划分 U-Mode 和 S-Mode，普通程序受限运行。
*   **主动通信**：程序通过准备寄存器并执行 `ecall`，主动陷入 OS 申请资源（系统调用）。
*   **被动打断**：异常（同步内部出错）或 中断（异步外部事件）会随时打断流水线。硬件通过**冲刷流水线**保证精确的现场。
*   **并发魔法**：OS 利用定时器中断，通过海量的 `lw` 和 `sw` 保存/恢复寄存器，实现进程的高速**上下文切换**。
*   **安全隔离**：最后，为了让多进程的内存在物理上不冲突，引出了下一步的课程重点：**虚拟内存的地址翻译**。
