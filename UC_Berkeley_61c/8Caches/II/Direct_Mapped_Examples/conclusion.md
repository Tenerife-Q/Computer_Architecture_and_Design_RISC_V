### CS61C Caches II：直接映射缓存 (Direct-Mapped) 终极精讲

**核心思想**：直接映射的本质是**硬件级别的哈希表（取模运算）**。主存中的每个地址，在 Cache 中有且只有一个合法的“车位”。

#### 一、 T.I.O 地址切分法与丹叔叔的矩形（对应图 4、图 6）
面对一个完整的 32 位内存地址，硬件会**从右向左（从小到大）**将其切分为三个“齿轮”：`[ Tag ] | [ Index ] | [ Offset ]`。

1. **Offset (小齿轮 / 块内偏移)**：
   * **作用**：定位 Block 内部的具体字节。
   * **公式**：$\text{位数} W = \log_2(\text{Block Size})$。这就构成了图 6 中矩形的 **WIDTH (宽)**，对应 $2^W$ 个物理字节。
2. **Index (中齿轮 / 缓存行索引)**：
   * **作用**：决定去查 Cache 的哪一行（Row）。
   * **公式**：$\text{位数} H = \log_2(\text{Cache 行数})$。构成了图 6 中矩形的 **HEIGHT (高)**，对应 $2^H$ 个物理行。
3. **Tag (大齿轮 / 身份标签)**：
   * **作用**：验证该行当前缓存的数据，到底是不是我们请求的主存块。
   * **精妙设计（对应图 5）**：**为什么 Tag 不存完整的 32 位地址？**
     * 省略 Offset：同一 Block 内的字节打包存取，共享 Tag。
     * **省略 Index（核心！）**：如果数据能停在 Index `10` 的车位里，说明它原始地址的中间段**必定是 `10`**。位置本身已经隐含了 Index 信息，无需重复存储浪费昂贵的硬件！

#### 二、 硬件查表“四步曲”（对应图 2）
当执行指令 `lw t0, 0(t1)` 时，CPU 将 32 位地址扔给 Cache：
1. **看 Index 定位**：瞬间直奔 Cache 对应的行。
2. **看 Valid Bit**：为 `0` 代表冷启动，直接 Miss；为 `1` 继续。
3. **比对 Tag**：提取 Cache 行里的 Tag，与地址前段的 Tag 对比。不匹配则 **Miss (Conflict Miss)**，匹配则 **Hit**！
4. **用 Offset 拿数据**：命中后，根据 Offset 提取 Block 中具体的字节给 CPU。

#### 三、 步长 (Stride) 效应的“二进制齿轮”模型（重点必考）
结合图 6 的“内存折叠”与底层的 `[T]|[I]|[O]` 二进制码（假设架构：8B 容量，2B 块，即 `[29位T] | [2位I] | [1位O]`），我们来看不同代码步长带来的极端影响：

**1. 步长 = 1 Byte（连续遍历）👉 空间局部性完美发挥**
* **二进制动作**：每次 `+1`，仅仅拨动**最右边的 Offset 小齿轮**（例如：`00|0` $\rightarrow$ `00|1`）。
* **结果**：读取地址 0 (Miss, 搬入0和1) $\rightarrow$ 读取地址 1 (**Hit**)。
* **命中率**：取决于块大小（这里是 50%，如果块大则是 63/64）。极佳！

**2. 步长 = Block Size（跨块遍历）👉 空间局部性失效**
* **二进制动作**：每次加 2（即二进制 `10`），**Offset 齿轮被彻底冻结（永远是0）**，直接转动 Index 中齿轮（例如：`00|0` $\rightarrow$ `01|0` $\rightarrow$ `10|0`）。
* **结果**：你完美的避开了 Cache 帮你打包搬进来的所有相邻数据，每次访问都在换行。
* **命中率**：**0%**。发生强制缺失 (Compulsory Miss)。

**3. 步长 = Cache Size（灾难步长）👉 终极抖动现象 (Thrashing)**
* **二进制动作**：每次加 8（即二进制 $1000$）。这个加法产生的进位，**直接跳过了底部的 I 和 O 齿轮，全部加在了 Tag 大齿轮上！**
* **结果**：地址变成 `000|00|0` $\rightarrow$ `001|00|0` $\rightarrow$ `010|00|0`。所有请求的 **Index 永远是 00**！它们像排队一样，带着不同的 Tag 去冲击同一个 Cache 行，互相顶替。
* **命中率**：绝对的 **0%**！发生极端的冲突缺失 (Conflict Miss)。

### 疑惑一：为什么地址是 32 位（bit），但 Cache 里面的一行（Block）却是 2 字节（Byte）？

**核心破解：门牌号的长度 $\neq$ 房间的大小！**

1. **什么是 32 位 (32-bit)？**
   在 RISC-V 32 位架构（RV32）中，**32位指的是“内存地址（门牌号）”的长度**。也就是说，CPU 发出的地址是用 32 根电线传过来的，比如 `0000...0000 1010`。
   **T.I.O（Tag 29位, Index 2位, Offset 1位）切分的，正是这 32 位的“门牌号”。**

2. **什么是 2 字节 (2 Bytes)？**
   这是 Cache 每一行用来**装真正数据（房间）**的容量。也就是那个 Block Size。

**为什么你会觉得违和？**
因为你提到了 *"RISC-V 的机器码"*。你的直觉是对的：一条 RISC-V 汇编指令（比如 `lw t0, 0(t1)`）翻译成机器码后，**本身就是 32 bits（也就是 4 Bytes）**！
如果 Cache 的一行只有 2 Bytes，那它连一条完整的 RISC-V 指令都装不下！

**真相大白**：
Dan 教授在这里使用的 `8B 总容量 / 2B 块大小` 的 Cache，是一个**纯粹为了方便在 PPT 上画出来的“微型玩具模型”**！
在真实的现代 CPU 中，Cache 的一个 Block 通常是 **64 字节 (64 Bytes)**，完全能装下 16 条连续的 RISC-V 指令。教授之所以用 2B，只是为了让你能用肉眼数清楚那几个格子，理解 T.I.O 的切分原理。**不要用真实的机器码大小去套这个玩具模型的容量。**

### 疑惑二：到底什么是“不存 Index 的原因”？（硬件视角的终极解释）

你对那个“酒店 204 房间”的比喻还有点懵，没关系，我们直接下沉到**物理硬件和电路**层面，这是最不讲道理但也最真实的逻辑。

假设我们依然是 4 行的 Cache（Index 分别是 `00`, `01`, `10`, `11`）。

**物理上的 Cache 是怎么建造的？**
在硅芯片上，Cache 是由实实在在的 4 排 SRAM 存储单元（格子）组成的。它们在物理位置上，从上到下排成了第 0 排、第 1 排、第 2 排、第 3 排。

* **主存数据的强绑定**：
  按照取模规则，地址中间两位是 `10` 的数据（比如地址 2, 6, 10），**硬件电路上被写死了，只能停进第 `2` 排的格子里**。

* **CPU 查找时的动作**：
  当 CPU 找地址 `000...000 | 10 | 0` (地址 2) 时。
  1. 硬件看到 Index 是 `10`。
  2. 硬件电路的开关直接连通了**物理上的第 2 排格子**。
  3. 此时，硬件只需把第 2 排格子里面存的 **Tag** 抽出来，跟地址前面的 `000...000` 相比对。

**为什么不需要在格子里存 `10` 这个 Index？**
因为硬件**此刻正在物理的第 2 排格子里翻找**！既然这个数据此时此刻乖乖地躺在第 2 排格子里，那它的原始地址中间必定是 `10`，否则它当初根本进不来这个格子。

**再换一个通俗比喻**：
这就像一个按姓氏首字母分类的文件柜。
你拉开标着 **"L"** 的抽屉，里面全姓李、林、刘。
为了区分具体是谁，文件袋上只需要写上他们的名字（**Tag**：比如“雷”、“华”）。
你**不需要**在文件袋上再写一遍“姓氏首字母是 L”（**Index**）。因为如果它不属于 L，它根本就不可能出现在这个抽屉里！你拉开 "L" 抽屉这个动作本身，就已经隐含了 Index 信息。

在芯片设计里，1 bit 都是极其昂贵的（占据物理面积，耗电）。既然“它停在第几行”就已经说明了它的 Index 是多少，那就绝对没必要把它存进去。

### 再次梳理你的复述，输出最终清晰版：

结合你上面的理解，我们把底层逻辑打通，输出一份严丝合缝的总结：

1. **宏观视角 (黑盒魔法)**：
   执行 `lw` 读内存时，CPU 发出一个 **32 bits 长**的二进制门牌号。Cache 作为中间商接管一切，若命中则极速返回数据；若未命中，Cache 负责去主存把包含该数据的**一整个 Block** 搬回来。

2. **地址解剖 (T.I.O 切分)**：
   硬件将这 32 位的门牌号（Address）切分成三个职能部分：
   * **O (Offset/块内偏移)**：决定读取搬进来的 Block 里的具体哪个 Byte。如果这个玩具模型的 Block 是 2 Bytes，那就需要 $2^1=2$，所以 Offset 占地址的最右 **1 bit**。
   * **I (Index/缓存行号)**：决定这个数据该停在 Cache 的哪一行。玩具 Cache 共有 4 行，需要 $2^2=4$，所以 Index 占中间的 **2 bits**。
   * **T (Tag/身份验证)**：地址的高位部分，在这个例子中是剩下的 **29 bits**。

3. **硬件级的极致“抠门”（省略 Index 存储）**：
   Cache 硬件的每一行，只存 1 位的 Valid 验证牌、29 位的 Tag 标签，以及 2 Bytes 的真实数据。
   **坚决不存那 2 位的 Index！** 因为 Cache 的物理行号（它位于芯片的第几排）本身就是 Index。数据只要停在了这排，就天然继承了这排的 Index 属性，存进去纯属浪费硅片面积。
