


你太牛了！这就到了 Cache 章节中最具实操性、也是考试中**必考大题**的环节：**缓存替换策略（Block Replacement Policy）以及 LRU 状态追踪。**

上一节课我们讲到，引入“包厢（Set）”之后，多个内存块终于可以和平共处了。但天下没有不散的筵席，**如果这个包厢的座位全都坐满了，而又有一个新人要挤进来，我们到底该把谁一脚踢出去（Evict）呢？**

这 4 张图，正是教你如何在这个残酷的“抢座游戏”中制定规则，并且手把手带你推演整个过程。我们一块儿来拆解：

---

### 第一步：为什么我们需要做选择？（图 1：Block Replacement Policy）

*   **直接映射 (Direct-Mapped)：没得选。** 
    尾号决定了你只能坐那个唯一的位子。如果有人，直接把他踢走，简单粗暴。这不需要什么策略。
*   **组相联 (N-Way) & 全相联 (Fully)：有得选。** 
    包厢里有 N 个座位。如果还有空座（Valid bit 为 0），新人直接坐下；**如果全坐满了（All are valid），选择困难症就犯了**。我们必须拿出一套规则来决定谁是那个倒霉的“牺牲者 (Victim)”。

---

### 第二步：三大踢人流派 (图 2：The Policies)

教授介绍了三种经典的替换策略：

1.  **LRU (Least Recently Used / 最近最少使用)：最聪明，也最累。**
    *   **核心思想：** 柿子捡软的捏。谁在最近这段时间里最没有存在感（最久没被访问过），就把谁踢走。
    *   **优点：** 完美契合了**时间局部性 (Temporal Locality)**。刚刚被访问过的数据，马上大概率还会被访问，所以要死死留在 Cache 里。
    *   **缺点（硬件的痛）：** 为了知道谁是“最老”的，硬件必须时刻记录每个块的访问顺序。对于 2-Way 来说很简单（只需 1 个 bit 记录即可）；但对于 8-Way 来说，排列组合状态极多，硬件开销和计算延迟会急剧上升！
2.  **FIFO (First In, First Out / 先进先出)：死板的官僚。**
    *   **核心思想：** 论资排辈。谁最早进来的，谁就先滚蛋。
    *   **缺点：** 它完全无视你在这期间被访问了多少次。即使你是一个每秒都被狂刷的“超级热点数据”，只要你排在最前面，时间一到照样被踢走。
3.  **Random (随机)：随缘。**
    *   **核心思想：** 闭着眼睛瞎选一个踢出去。
    *   **为什么它还有用？** 当你的工作负载（Workload）本身就没有什么时间局部性时，不管你用多复杂的 LRU，命中率都不高。这时候，造价极低、速度极快的 Random 策略反而成了一个高性价比的设计选择。

---

### 第三步：硬核推演！LRU 状态机走读 (图 3：Block Replacement Example)

**这绝对是期末考试必考的题型（Cache Trace）。**
请看图 3 右侧的那一列状态图，这是一个 **2-Way（2路组相联）**，共有 **2个包厢（Set 0, Set 1）** 的 Cache。
我们跟着教授，走一遍访问序列：`0, 2, 0, 1, 4, 0`。

> ⚠️ **看图指南：** 注意那个橙色的 **`lru`** 标记。它就像一张**“死神黑名单”**，指向谁，下一次踢人的时候就踢谁！

1.  **读 `0` (Miss)：** 进 Set 0，坐左边。右边空着，所以 `lru` 标记打在右边（或者可以理解为：空位优先）。
2.  **读 `2` (Miss)：** 进 Set 0，坐右边。此时 Set 0 坐满了！因为 `0` 是先来的，`2` 是刚刚访问的。所以相对而言，`0` 变成了“最久未使用的”，**`lru` 标记转移到了 `0` 的头上**。
3.  **读 `0` (Hit!)：** 关键来了！CPU 再次访问了 `0`。`0` 瞬间变成了“全场最热”的数据！于是，旁边的 `2` 自动沦落为“最久未使用”，**`lru` 标记立刻转移到了 `2` 的头上**。
4.  **读 `1` (Miss)：** 进 Set 1，坐左边。目前互不影响。
5.  **读 `4` (Miss & Replace！)：** `4` 映射到了 Set 0。但此时 Set 0 已经满了（装着 `0` 和 `2`）。该踢谁？
    回头看第 3 步，**`lru` 标记在 `2` 头上！** 所以硬件毫不留情地把 `2` 踢走，把 `4` 填了进去。
    填完后，因为 `4` 是新来的，`0` 变成了老的，所以 `lru` 标记又回到了 `0` 的头上。
6.  **读 `0` (Hit!)：** `0` 依然坚挺在 Cache 里，完美命中！

**🔥 教授的激动时刻（看图 3 底部字幕）：**
教授说：“在 0 和 4 之间我能愉快地复制，这在直接映射中是做不到的。”
如果是之前的直接映射，`0`, `2`, `4` 全部都会抢同一个位置，命中率将是惨烈的 0%。但现在有了 **2-Way**，`0` 和 `4` 可以在同一个包厢里面对面坐着，互相不打架！这就叫技术的进步！

---

### 第四步：Cache Simulator 验证 3C (图 4)

教授最后展示了一个牛逼的网页版缓存模拟器，把刚才的序列输进去验证。我们重点看右边的统计数据和颜色：

*   **Compulsory Miss (强制未命中 / 蓝色) = 3 次。** 
    序列中第一次出现的 `0`, `2`, `1`，因为 Cache 刚开机啥也没有，这 3 次必然是冷启动 Miss。
*   **Conflict Miss (冲突未命中 / 橙色) = 1 次。**
    这 1 次是谁贡献的？就是我们刚才强行踢掉 `2` 换进来的那个 **`4`**！
    **灵魂拷问：为什么它是 Conflict 而不是 Capacity (容量)？**
    因为在读取 `4` 的那一刻，Cache 的总容量是 4 个块，而此时 Cache 里只有 `0`, `2`, `1`（共 3 个块）。**整个 Cache 并没有满（Set 1 还有个空位呢）！** 它只是倒霉地又撞进了人满为患的 Set 0，所以这叫 Conflict Miss。
*   **Hit Rate (命中率)：** 2次命中 / 6次总访问 = 33.33%。

### 给你的复习小结：
看完这四张图，你需要建立这样的条件反射：
1.  一听到 **Direct Mapped** -> 脑补出“死板排座，天天冲突打架”。
2.  一听到 **2-Way Set Assoc** -> 脑补出“2人一桌，缓解冲突”。
3.  一遇到 **LRU** 题目 -> 立刻在纸上画格子，每次 Hit 后，都要**强行把 LRU 标记（或者叫靶心）移到另一个没被访问的倒霉蛋头上**。

这一段逻辑极其严密，是拿来出计算题和 Trace 题的绝佳素材。你对图 3 里 `lru` 标记跳来跳去的逻辑完全吃透了吗？如果还有一丝疑惑，我们可以自己编一个序列再手工推演一遍！
